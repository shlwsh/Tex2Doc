## 6.2 当前阻塞

### 6.2.1 现象

`cargo test -p doc-tex-facade` 在 **Windows**（MiKTeX 探测到本机有 xelatex）上 6 次重试均以 `exit_code=4294967295`（-1，外部 kill）结束。关键日志片段（来自 `terminals/497233.txt` / `76943.txt`）：

```text
running 5 tests
test compile_paper3_bad_tex_returns_error ... ignored
test compile_paper3_cache_hit ... ignored
test compile_paper3_with_xelatex ... ignored
test facade_engine_unavailable_for_preferred ... ok
test facade_with_unavailable_engine_returns_no_engine has been running for over 60 seconds
```

**无 `test result:` 汇总行**——cargo test 进程被外部 kill 时还没跑完第 5 项。

### 6.2.2 关键观察

1. **5 个测试中 4 个干净**——3 个 `#[ignore]` 不跑；第 4 项 `facade_engine_unavailable_for_preferred` `ok`。
2. **第 5 项函数体是纯同步的**（详见下面 §6.3 分析），**没有**任何 I/O、async、进程拉起、`Cache` 写盘——按理应当 < 1 ms 返回。
3. **lib + 其他三个 test binary 全过**（来自 `terminals/743291.txt`）：
   ```text
   test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
   test result: ok. 3 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
   test result: ok. 6 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
   ```
   唯一挂的 binary 就是 `tests/compile.rs`。
4. **10 次重试**（杀掉僵尸进程 / 加 is_available 超时 / 单独跑这个 binary）都看到同一行为：第 4 项 `ok` → 第 5 项卡住。

### 6.2.3 复现命令

```bash
cargo test -p doc-tex-facade --test compile 2>&1 | head -20
```

跑出来的最后一行永远是 `... has been running for over 60 seconds`，**不会**输出 `test result:`。

## 6.4 根因分析（待验证）

### 6.4.1 第 5 项测试的静态审视

[`tests/compile.rs:60-74`](../../../crates/tex-facade/tests/compile.rs)：

```rust
#[test]
fn facade_with_unavailable_engine_returns_no_engine() {
    let project = TexProject::from_main(paper3_main());
    let unavailable: Arc<dyn TexBackend> = Arc::new(FakeBackend {
        kind: EngineKind::Xelatex,
        available: false,
    });
    let facade = TexFacade::with_backend(unavailable, &project).with_concurrency(1);
    let engines = facade.available_engines();
    assert_eq!(engines, vec![EngineKind::Xelatex], "应保留 fake 引擎");
    eprintln!("✓ fake engine facade OK：engines={engines:?}");
}
```

逐行性质：

| 行 | 调用 | 是否触碰 I/O / 异步 / 进程 |
|---|------|--------------------------|
| 1 | `paper3_main()` | ❌ 否（`env!("CARGO_MANIFEST_DIR")` + `PathBuf::join`） |
| 2 | `TexProject::from_main(...)` | ❌ 否（仅 path 计算） |
| 3 | `Arc::new(FakeBackend { ... })` | ❌ 否 |
| 4 | `TexFacade::with_backend(...)` | ❌ 否（构造 `vec![b]` + `Cache::for_workdir`（仅 path 计算）+ `Semaphore::new(2)`） |
| 5 | `.with_concurrency(1)` | ❌ 否（`Semaphore::new(1)`） |
| 6 | `facade.available_engines()` | ❌ 否（`iter().map(\|b\| b.kind()).collect()`） |
| 7 | `assert_eq!` | ❌ 否 |
| 8 | `eprintln!` | ❌ 否（向 stderr flush） |

**结论：第 5 项函数体本身不可能花 60+ 秒。**「`has been running for over 60 seconds`」一定是 **第 5 项函数返回了，但 cargo test 没看到它返回**——即整个 test binary 卡在了某处。

### 6.4.2 推测：tokio `Runtime::drop()` 在 Windows 上阻塞主线程

第 4 项 `facade_engine_unavailable_for_preferred` 创建了 **multi-thread tokio runtime**：

```rust
let rt = tokio::runtime::Runtime::new().unwrap();
let result = rt.block_on(facade.compile_to_pdf(&project));
// ...
// rt 在 test 结束时 drop
```

虽然 `compile_to_pdf` 在 `pick_backend` 阶段就返回 `Err`（**没有真正进入 async runtime**），但 `Runtime::drop()` 仍会向所有 worker 线程发 shutdown 信号并 `join()`。在 Windows 上，这个 join 可能阻塞数秒甚至数十秒——尤其是当 `block_on` 的 future 没有真正 yield 过一次时（worker 线程从未被调度，可能尚未到达"可被取消"的点）。

**若此推测正确**，修复办法是第 4 项改用 `current_thread` runtime：

```rust
let rt = tokio::runtime::Builder::new_current_thread()
    .enable_all()
    .build()
    .unwrap();
```

`current_thread` runtime 没有 worker 线程，`drop()` 几乎是瞬时的。

### 6.4.3 备选推测（待排除）

- **(a) `Cargo.lock` 改动**引入新的 transitive dep（如 `chrono 0.4` 的本地化）导致 crate 加载慢——但 `cache_key.rs` 6 项全过，crate 本身能加载。
- **(b) Windows 杀进程时信号传播**——`>60s` 警告触发后，task runner 主动 kill `cargo`，因此 `test result:` 永远写不出来。
- **(c) `eprintln!` 的 stderr 在 cargo 测试缓冲下刷不出来**——不会让 test "卡 60s"，只会让用户看不到日志。

## 6.5 建议的下一步（按优先级）

1. **【最小改动】** 把 `facade_engine_unavailable_for_preferred` 里的 `tokio::runtime::Runtime::new()` 改为 `Builder::new_current_thread().enable_all().build().unwrap()`；重跑 `cargo test -p doc-tex-facade --test compile`。
   - **预期**：第 5 项立即返回；`test result: ok. 2 passed; 0 failed; 3 ignored`。
   - **若仍卡**：根因不是 (a) 这条，继续看 (b) / (c)。
2. **【若 1 仍卡】** 在第 5 项开头加 `eprintln!("ENTER test5")` + 结尾 `eprintln!("EXIT test5")` + `--nocapture` 重跑；看 cargo 实际收到了哪行。
3. **【若 1 仍卡】** 在第 4 项 `Runtime::drop()` 前后各加一行 `eprintln!`，看 drop 是否真在 60s 内返回。
4. **【都不奏效】** 把第 5 项改成显式 `eprintln!` 4 步：构造前、构造后、`available_engines` 前、assert 前，定位到底在哪一步看起来"卡住"（实际上不可能是它自己卡住，但能确认 cargo 是怎么统计的）。
