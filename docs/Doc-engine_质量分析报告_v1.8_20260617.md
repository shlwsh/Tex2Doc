# Tex2Doc 转换引擎质量分析报告

**版本**: v1.8
**日期**: 2026-06-17
**状态**: 内部分析 — 机密

---

## 一、概述与背景

本报告对 Tex2Doc V0.2.0 转换引擎在 `paper3`（软件学报论文）上的输出质量进行系统性诊断，通过与原生 TeX→PDF（Oracle PDF）逐页字符数、内容结构、格式三个维度的对比，分析工作结果质量差的根本原因，并评估各风险点的严重程度。

**诊断方法**:
- 逐页 `pdftotext` 字符数对比（layout 模式）
- 语义 AST 块结构统计（来自 `paper3_e2e` 集成测试）
- LaTeX 源码与 AST 对比分析
- DOCX zip 结构检查（`unzip -l`）
- PDF 文件大小交叉验证

## 三、质量问题根因分析

### 3.1 【🔴 P0 — 阻断性】soffice 完全无法渲染 DOCX（图片 + 文档结构）

**症状**:
- V2 DOCX 在 `soffice --headless --convert-to pdf` 时，要么报告 `Error: source file could not be loaded`，要么生成无图片的空白 PDF。
- 将 V2 的 `document.xml` 注入 python-docx 容器后，同样报 `source file could not be loaded`。
- `soffice` 生成的 PDF 中，图片完全缺失。

**根因**: `soffice`（LibreOffice）的 DOCX 解析器对 Tex2Doc 生成的 XML 结构存在根本性不兼容。经过多轮调试（EMU 单位修正、`graphicFrameLocks` 补充、命名空间 URI 纠正、PNG→JPEG 转换），问题仍未解决。这表明可能是以下因素的组合：
1. 自定义段落样式（`w:pStyle`）与 soffice 样式系统冲突
2. 复杂的 `w:rPr`（run properties）嵌套结构不兼容
3. Header/footer 中的动态字段（`{{PAGE}}`）可能引发解析异常
4. 某些 LaTeX 宏展开后的 XML 实体编码异常

**风险**: 若不能修复，Tex2Doc 的 PDF 输出通路**完全失效**，只能依赖 Word/LibreOffice Writer 的手动另存 PDF 功能。

---

### 3.2 【🔴 P0 — 阻断性】图片未嵌入 DOCX（`word/media/` 目录缺失）

**症状**:
- `unzip -l main-jos.docx` 显示 **无** `word/media/` 目录
- DOCX 文件仅 26 KB（不含图片的纯文本包）
- 生成的 PDF 中图片完全缺失
- 8 个 `Block::Figure` 在 AST 中存在，但 `word/media/image{N}.jpg` 未被打包

**根因**: `docx-writer` 中虽然实现了 `EmbeddedImage` 收集和 `packer.rs` 的 `word/media/` 写入逻辑，但**实际代码路径从未触发**。可能原因：
1. `serializer.rs` 中 `Block::Figure` 的图片嵌入逻辑有条件分支未满足（如 `image_assets` 为 `None` 或 `path` 不匹配）
2. `packer.rs` 的 `embedded_images` 参数传递链断裂
3. `paper3_e2e` 测试调用的 `pack_with_page_setup` 未传入 `image_assets`，导致图片字节从未提供

**风险**: 图片不嵌入意味着任何包含图片的论文/报告都无法正确转换，这是**最基础的功能缺失**。

---

### 3.3 【🔴 P0 — 内容性】算法环境（algorithm）渲染为原始 LaTeX

**症状**:
- Oracle p6 包含完整的 `Algorithm 1` 伪代码（带行号、缩进、关键词高亮）
- V2 中 `algorithm` 环境被识别为 `RawFallback`，渲染为原始 LaTeX 代码片段（含 `\ForEach`、`\If`、`\tcp*` 等命令）
- 段落 P29 仅包含"上传失败时，第 n 次重试前的等待时间定义为"一句话，缺失算法伪代码主体

**根因**: `latex-reader` 的 `lower_environment` 中没有 `algorithm` / `algorithm2e` / `algorithmicx` 环境的专门处理分支。当前 fallback 策略是将整个环境内容作为 `RawFallback` 块输出，完全没有做语义解析。

**影响内容**:
- 缺失算法伪代码（约 500–1000 字符/次）
- 缺失 `\KwIn`/`\KwOut` 等关键字语义化
- 缺失行号、缩进等结构信息

**风险**: 任何含算法伪代码的学术论文都无法正确渲染，这是 paper3 缺失内容的最大单一来源。

---

### 3.4 【🟡 P1 — 内容性】公式渲染存在 LaTeX 残渗

**症状**:
- Oracle: `d₀ = 200 ms`（下标为真正的 Unicode 字符）
- V2: `d_=200,ms`（LaTeX 风格下标 `$d_$`，逗号被插入）

**根因**: `latex_to_text` 对某些数学上下标的处理不完整。具体来说：
- `$d_$` 中的下标 `_` 被保留为普通文本，未转换为 Unicode 字符或下标格式
- `$n{=}3$` 中的 `{=}` 被渲染为字面量 `=`，Oracle 中为 `$n=3$`
- `$\pm$` 在 V2 中正常渲染为 `±`，但 `$d_` 系列有问题

**影响**: 中文学术论文中的数学符号渲染质量受损，影响可读性。

---

### 3.5 【🟡 P1 — 内容性】交叉引用（`\ref`、`\pageref`）未解析

**症状**:
- V2 段落文本中出现 `表 tab:related_compare`、`图 fig:compare`、`表 tab:tech` 等字面引用
- Oracle 中这些引用被解析为"表 1"、"图 2"、"表 4"等实际编号

**根因**: `latex-reader` 的宏展开阶段将 `\ref{tab:xxx}` 展开为 `\ref{tab:xxx}`（字面量），`latex_to_text` 没有将其转换为编号。引用解析需要在 lowering 阶段维护一个 label→number 的映射表，并在文本序列化时做替换。

**影响**: 学术论文中的交叉引用全部失效，文档引用完整性归零。

---

### 3.6 【🟡 P1 — 内容性】表格环境渲染缺陷

**症状**:
- V2 p4 显示表头后，内容出现两次（第一次在表头后，第二次在页面底部）
- Oracle p4 中表格正确占据整个页面，表头 + 8 行数据
- V2 p8 出现原始 LaTeX 表格代码（`{}{@{}ll}`）

**根因**:
1. `flushleft` 环境中的表格被部分处理，内容重复渲染
2. `latex-reader` 对 `\begin{tabular}` 环境的解析不完整，产生了错误的 AST 结构
3. `docx-writer` 的 `write_table` 对复杂表格（多列、复杂列定义）处理有缺陷

**影响**: 表格是学术论文最重要的结构之一，渲染错误直接损害文档可用性。

---

### 3.7 【🟡 P1 — 内容性】页内内容分布严重不均

**症状**:
- Oracle 的内容分布在 16 页中较为均匀
- V2 将内容压缩到 12 页，导致同一页包含不同章节的内容（Oracle 分属不同页）
- 页 9（V2）字符数比 Oracle 多 65%，说明字体大小、行距或段间距设置有问题

**根因**: `docx-writer` 的样式系统（字号、行距、段间距）与 Oracle 的 JOS 模板差异较大：
- JOS 模板使用 5 号字（约 10.5pt），V2 可能使用了不同的基础字号
- 行距设置可能偏小
- 段落间距（段前/段后）设置偏紧

---

### 3.8 【🟠 P2 — 格式性】摘要中英双语内容重复

**症状**:
- V2 p1 的摘要部分包含了完整的中英文摘要 + 关键词 + 英文引用格式 + 英文标题/作者/摘要/关键词
- 这些内容在 Oracle 中出现在 p1（中文部分）和 p2 开头（英文部分）
- V2 在 p1 末尾附加了"摘 要\n收稿时间..."等元信息，Oracle 中无此内容

**根因**: 前后言（front matter）提取逻辑未正确去重，将所有语言版本的前言内容全部串接输出。

---

### 3.9 【🟠 P2 — 格式性】Greek 字母 α、β 等渲染正常，但间距不佳

**症状**:
- V2 p6 段落 P25 中 `α` 和 `β` 渲染正确
- 但 `α · H[p].count` 中点号 `.` 被渲染为 `·`，而 Oracle 中可能是数学乘号

**评估**: 属于次要格式问题，不影响核心内容。

## 五、根本原因总结

### 5.1 架构层面的问题

```
Tex2Doc 转换流程中存在三个"最后一公里"断裂：

  LaTeX 源码
      ↓
  [latex-reader] 语义 AST（Document）
      ↓  ⚠️ 断裂点 A: algorithm/tabular/equation 特殊环境未语义化
  [semantic-ast] Document（理想状态）
      ↓
  [docx-writer] DOCX XML（理想状态）
      ↓  ⚠️ 断裂点 B: 图片字节未传入 image_assets
  [packer] .docx ZIP（含 word/media/）
      ↓  ⚠️ 断裂点 C: soffice 与我们 XML 的不兼容
  [soffice] .pdf（实际：图片全丢/甚至无法加载）
```

### 5.2 核心矛盾

**矛盾一：结构解析 vs. 格式渲染**
- Tex2Doc 的核心优势在于语义 AST 的结构化表示
- 但 `algorithm`、`tabular`、`\ref` 等 LaTeX 特性没有对应的 AST 节点
- fallback 策略（RawFallback）只是转储原始文本，完全破坏了文档结构

**矛盾二：OOXML 规范符合性 vs. soffice 兼容性**
- Tex2Doc 严格按照 OOXML 规范生成 XML
- 但 soffice（LibreOffice）的实现存在偏离规范的地方，对某些结构有隐性依赖
- python-docx 能工作的原因在于它生成的 XML恰好命中了 soffice 的兼容路径

**矛盾三：内容正确性 vs. 视觉一致性**
- 目前关注点仍在"内容是否完整"
- 但即使内容完整，视觉一致性（页面布局、字体、间距）仍无法保证
- 两个目标同时满足需要更精细的样式映射和测试机制

## 七、结论

Tex2Doc 当前 V0.2.0 版本在**结构解析**（章节、段落、列表识别）和**基础样式**（字体、颜色、标题层次）上已具备一定能力，但在**内容完整性**上存在三个 P0 阻断性问题：

1. **图片完全缺失**（图片未嵌入 DOCX）
2. **soffice 无法渲染**（PDF 输出失效）
3. **算法环境渲染为原始 LaTeX**（内容实质性丢失）

这三项是当前质量差的**最主要原因**，三者合计导致了约 29% 的字符缺失和 4 页内容完全丢失。

根本原因在于：Tex2Doc 的设计理念（语义 AST → 结构化 DOCX）与 LaTeX 的实际复杂性之间存在较大鸿沟——LaTeX 的特殊环境（algorithm、tabular）和 soffice 的兼容性要求都需要更精细的处理。修复工作应按 P0 → P1 → P2 的顺序推进，优先恢复图片和 PDF 输出能力，再完善内容质量和测试体系。
