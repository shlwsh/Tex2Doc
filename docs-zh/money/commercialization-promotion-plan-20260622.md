# Tex2Doc 商业化推广缺口分析与规划方案
> **版本 / Version**: v2.0
> **最后更新日期 / Last Updated**: 2026-06-26



**日期**：2026-06-22
**范围**：Tex2Doc / Semantic TeX Engine 的商业化推广、产品包装、销售转化、发布运营与商业支撑能力。
**结论**：当前项目已经具备商业化 Preview 原型和较强技术资产，但还不适合直接公开自助收费。最合适路径是先做邀请制 Beta，验证质量、需求和价格，再进入付费 Beta，最后发布 Pro Desktop + Cloud GA。

## 一、当前可用于商业化的基础

| 资产 | 当前状态 | 可用于推广的价值 |
| --- | --- | --- |
| LaTeX 到 DOCX 转换核心 | 已有 Rust 核心、Semantic TeX Engine、DOCX writer、质量报告 | 可以主打本地高保真、Word 可编辑、学术论文场景 |
| Profile / RuleEngine / QualityGate | P0-P4 已完成，P5-P9 持续推进 | 可以包装为“面向期刊模板的智能转换” |
| 桌面客户端 | Slint Desktop 已具备本地转换、云转换、账号、账单、recent jobs、诊断包、更新检查 preview | 可以作为首个商业交付载体 |
| 商业 API 与云 Worker | 已有 preview API、内存态 worker、上传 zip 生成 DOCX/report | 可以支持内测客户的云端试用 |
| 回归验证 | paper3、realistic fixture、nightly regression、Word 打开验证已启动 | 可以逐步形成质量证明和销售素材 |
| 多端入口 | CLI、桌面端、Web/PWA、Chrome 扩展文档和工程基础存在 | 可以构建从免费入口到付费桌面的获客漏斗 |

## 二、商业化推广仍欠缺的工作

### 2.1 产品定位与卖点包装

当前技术文档丰富，但还缺面向购买者的产品表达。

| 欠缺项 | 影响 | 建议安排 |
| --- | --- | --- |
| 清晰 ICP | 不知道优先卖给谁，推广内容容易发散 | 先锁定 3 类人群：研究生/论文作者、实验室与课题组、期刊/出版社技术编辑 |
| 一句话定位 | 用户无法快速判断是否相关 | 定位为“把真实 LaTeX 论文项目转换为可编辑 Word DOCX 的桌面与云端工具” |
| 核心卖点页 | 不能承接外部流量 | 制作 landing page、GitHub README 商业区、中文介绍页 |
| 对比材料 | 难以解释为什么不用 pandoc 或手工改 Word | 输出 pandoc / Tex2Doc / 手工改稿对比表，强调模板、公式、引用、图片、报告 |
| Demo 素材 | 缺少可传播内容 | 准备 3 个示例：英文期刊、中文论文、复杂公式/图片论文 |

### 2.2 可购买产品形态

当前更像技术原型，还缺“用户买到什么”的清晰边界。

| 产品形态 | 建议状态 | 说明 |
| --- | --- | --- |
| Free Preview | 立即准备 | 免费试用，限制文件大小、次数、云端队列优先级 |
| Pro Desktop | 付费 Beta 主力 | 本地转换、批量任务、质量报告、云端额度、自动更新 |
| Cloud Credits | Beta 后期准备 | 按转换次数或页数计费，适合临时用户 |
| Team / Lab | 付费 Beta 后验证 | 课题组共享额度、多人账号、发票、统一管理 |
| Enterprise | GA 后推进 | 私有化部署、数据留存策略、SLA、合同支持 |

### 2.3 定价与收入模型

当前已有账单入口和 plans preview，但缺真实价格、套餐边界和支付闭环。

| 欠缺项 | 建议方案 | 验收标准 |
| --- | --- | --- |
| 套餐定义 | Free、Pro、Team、Enterprise 四层 | 每层写清本地转换、云转换、文件大小、支持级别 |
| 价格实验 | Pro 先测月付和年付，Team 按席位或共享额度 | 至少完成 20 个意向用户访谈或问卷 |
| 支付闭环 | Stripe 或国内支付二选一先通一条 | checkout、portal、退款、发票流程跑通 |
| 用量账本 | 记录转换次数、文件大小、云端时长、失败退额 | 用户可在桌面端看到额度变化 |
| 免费试用策略 | 免费 5 次云转换或 7 天 Pro | 能追踪试用到激活、激活到付费 |

### 2.4 推广渠道与获客资产

当前缺少增长渠道设计，不能只依赖代码仓库自然曝光。

| 渠道 | 首批动作 | 产出 |
| --- | --- | --- |
| SEO 内容 | 写 6 篇长尾文章，如“LaTeX 转 Word 保留公式” | 中文站、英文站、GitHub Pages 或产品博客 |
| GitHub | 优化 README、Release、示例 GIF、issue 模板 | 把开源访问转成 waitlist |
| Chrome Web Store | 准备扩展上架素材和隐私说明 | 作为轻量获客入口，引导下载桌面端 |
| 学术社区 | 小红书/知乎/B站/ResearchGate/Reddit 选 2 个先做 | 每周 2 条内容，收集试用名单 |
| 高校/实验室 | 建 50 个目标实验室名单 | 邮件触达、微信群试用、访谈记录 |
| 期刊/出版社 | 建 20 个技术编辑目标名单 | 验证 B2B 场景与定制需求 |
| 合作渠道 | 找论文润色、排版、投稿服务商 | 试点转介绍或联合服务 |

### 2.5 质量证明与可信素材

商业推广必须让用户相信“复杂论文真的能转”。

| 欠缺项 | 补齐工作 | 验收标准 |
| --- | --- | --- |
| 公开质量基准 | 建立真实样本集和脱敏样本集 | 至少 30 个真实项目，覆盖英文、中文、公式、图片、Bib |
| Word 打开验证 | 把 Word/LibreOffice 打开验证纳入发布门禁 | Beta 发布包必须通过样本集检查 |
| 对比报告 | 每个示例输出 Tex2Doc 与 pandoc 对比 | 可截图、可下载、可复现 |
| 失败分类 | 归类为语法不支持、资源缺失、模板冲突、字体问题 | 桌面端和报告都能给用户可读建议 |
| 客户案例 | 形成 3 个 beta case study | 包含原始痛点、转换效果、节省时间 |

### 2.6 发布交付与安装体验

当前桌面端有 preview 能力，但商业推广需要稳定可下载、可更新、可恢复。

| 欠缺项 | 建议安排 | 验收标准 |
| --- | --- | --- |
| Windows 安装包 | 优先做 Windows x64 installer | 用户可下载安装、卸载、升级 |
| macOS / Linux 包 | Beta 后补齐 | 至少提供签名/校验说明和手工安装路径 |
| 代码签名 | Windows 证书、macOS notarization | 下载不被系统明显拦截 |
| 自动更新 | manifest、下载、验签、安装执行、回滚 | Beta 包可从旧版本升级到新版本 |
| 诊断包 | 一键导出日志、配置、转换报告 | 支持人员可定位 80% 常见失败 |
| Release Notes | 每次发布写用户可读变更 | 用户知道是否值得升级 |

### 2.7 云端生产能力

当前云端 worker 是 preview，不足以承接公开流量和付费承诺。

| 欠缺项 | 补齐工作 | 验收标准 |
| --- | --- | --- |
| 生产 auth | JWT、refresh token、密码哈希、会话撤销 | 通过基础安全测试 |
| 数据持久化 | PostgreSQL 保存用户、订阅、任务、用量 | 服务重启后任务和账本不丢 |
| 对象存储 | 上传 zip、DOCX、report、日志分层保存 | 可配置过期删除 |
| Worker sandbox | 限制 CPU、内存、时间、文件大小 | 恶意或异常项目不会拖垮服务 |
| 队列系统 | Redis / Postgres queue / managed queue | 高峰时任务可排队、重试、取消 |
| 监控告警 | 请求、错误、队列、转换耗时、成本 | 出现高错误率能告警 |
| 数据合规 | 隐私政策、删除机制、保留周期 | 用户能请求删除上传文件 |

### 2.8 销售与客户成功

目前缺的是把试用者变成付费客户的流程。

| 欠缺项 | 建议动作 | 产出 |
| --- | --- | --- |
| Waitlist | landing page 收集邮箱、身份、论文场景 | 每周导出线索表 |
| CRM | 用轻量表格或 Notion 先管理 | 记录来源、需求、试用状态、付费意向 |
| Onboarding | 10 分钟试用指南、示例项目、常见问题 | 降低首次转换失败率 |
| Support | 建邮箱、微信群、GitHub issue 分流规则 | 48 小时内响应 beta 用户 |
| 销售材料 | 一页纸 PDF、演示视频、FAQ、报价单 | 支持高校/实验室采购沟通 |
| 用户反馈闭环 | 每个失败样本进入质量 backlog | 每周输出质量 Top 10 问题 |

### 2.9 法务、合规与品牌

开源许可不等于商业服务合规，推广前需要补齐基础文件。

| 欠缺项 | 说明 |
| --- | --- |
| 产品名称与域名 | 统一 Tex2Doc / Semantic TeX Engine 的对外名称，注册域名 |
| 商标风险检查 | 避免与已有产品重名或侵权 |
| 隐私政策 | 说明上传文件、日志、账号、支付数据如何处理 |
| 服务条款 | 明确可用性、退款、数据删除、禁止上传内容 |
| 开源许可清单 | 梳理 MIT/Apache-2.0 及依赖许可证 |
| 数据处理协议 | 面向 Team / Enterprise 准备 DPA 草案 |
| 安全说明 | 说明本地转换、云转换、数据保留和删除策略 |

## 三、12 周推进计划

### 第 0-1 周：商业化基础包

目标：能解释产品、收集线索、邀请第一批用户。

| 编号 | 工作 | 负责人角色 | 交付物 |
| --- | --- | --- | --- |
| W1-1 | 明确 ICP、定位、竞品对比 | 产品/市场 | `ICP + positioning` 文档 |
| W1-2 | 建 landing page 或 GitHub Pages | 前端/市场 | 首页、waitlist 表单、隐私声明草案 |
| W1-3 | 准备 3 个可下载 demo | 产品/研发 | 原始 LaTeX、输出 DOCX、质量报告、截图 |
| W1-4 | 定义 Free / Pro / Team 套餐草案 | 产品/商业 | 套餐表、试用规则、价格实验表 |
| W1-5 | 建线索表和反馈模板 | 运营 | CRM 表、访谈问题、失败样本收集表 |

验收：可以向 10 个目标用户发送试用邀请，并能承接报名。

### 第 2-3 周：邀请制 Beta

目标：让真实用户跑通转换，收集质量和付费信号。

| 编号 | 工作 | 负责人角色 | 交付物 |
| --- | --- | --- | --- |
| W2-1 | 发布 Windows Preview 安装包 | 研发/发布 | installer、checksum、release notes |
| W2-2 | 完成首批 10-20 位用户试用 | 运营/产品 | 用户访谈记录、转化漏斗 |
| W2-3 | 建真实样本质量库 | 研发/质量 | 脱敏样本、质量指标、失败分类 |
| W2-4 | 完善诊断包和支持流程 | 研发/客服 | 诊断包说明、support playbook |
| W2-5 | 输出首版案例文章 | 市场 | 1 篇技术案例、1 篇用户场景文章 |

验收：至少 10 个真实项目完成转换，形成 Top 10 质量问题和 Top 5 购买理由。

### 第 4-6 周：付费 Beta 准备

目标：打通支付、用量、云端持久化和可恢复交付。

| 编号 | 工作 | 负责人角色 | 交付物 |
| --- | --- | --- | --- |

| W4-2 | 上线生产级账号与用量账本 | 后端 | PostgreSQL schema、quota ledger、审计日志 |
| W4-3 | 云 Worker 持久化与对象存储 | 后端/运维 | queue、artifact storage、过期清理 |
| W4-4 | 发布自动更新闭环 | 桌面/发布 | 下载、验签、安装、回滚 |
| W4-5 | 上线质量 dashboard | 质量/研发 | profile 维度通过率、失败率、耗时 |
| W4-6 | 准备报价单和发票流程 | 商业/财务 | Pro/Team 报价单、付款说明 |

验收：可以让 3-5 个早期用户真实付款或完成付款模拟，并正确扣减云端额度。

### 第 7-9 周：公开 Beta 与渠道放量

目标：从人工邀请转向可控公开获客。

### 第 10-12 周：GA 决策

目标：判断是否可以正式发布 Pro Desktop + Cloud GA。

| 编号 | 工作 | 负责人角色 | 交付物 |
| --- | --- | --- | --- |
| W10-1 | GA 发布门禁评审 | 产品/研发/质量 | GA readiness checklist |
| W10-2 | 安全与合规审查 | 后端/法务 | 隐私、条款、数据删除、依赖许可证清单 |
| W10-3 | 支持与 SLA 定义 | 运营/商业 | 支持级别、响应时间、退款政策 |

验收：质量、支付、支持、合规、监控全部达标后再宣布 GA。

## 六、主要风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 转换质量不稳定 | 复杂模板、公式、引用、中文排版失败 | 先限制支持范围，建立失败样本库和 profile 质量 dashboard |
| 推广过早导致口碑损失 | 用户下载后无法完成真实论文转换 | 先邀请制 Beta，人工支持，不公开承诺全量模板 |
| 云端成本不可控 | 大文件、恶意输入、长时间任务拖垮 worker | 文件大小限制、sandbox、队列、超时、额度扣减 |
| 支付和合规滞后 | 用户想付费但无法购买或不信任上传 | 先做桌面本地 Pro，再逐步开放云端付费 |
| 客户样本涉密 | 论文未发表，不愿上传云端 | 强调本地转换，云端设置保留周期和删除机制 |
| 多端摊薄资源 | 桌面、PWA、扩展、云端同时推进 | 商业主线先押 Windows Desktop + Cloud，其他端做获客入口 |

## 八、最终建议

不要把当前目标定义为“立刻公开商业发布”。更稳的目标是：

1. 用 2 周完成商业化基础包和邀请制 Beta。
2. 用 4-6 周验证真实论文质量、定价和付费意愿。
3. 用 8-12 周决定是否进入公开 Beta 或 GA。

推广话术要聚焦一个明确承诺：Tex2Doc 不是泛用文档转换器，而是面向真实 LaTeX 论文项目的 DOCX 转换工作台，优先解决公式、引用、图片、期刊模板和质量报告这些学术用户最痛的环节。
