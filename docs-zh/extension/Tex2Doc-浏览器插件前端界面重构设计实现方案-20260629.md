# Tex2Doc 浏览器插件前端界面重构设计实现方案

> 版本：v1.0  
> 日期：2026-06-29  
> 范围：`apps/browser-extension` 的 `popup`、`sidepanel`、`options` 三个前端入口  
> 目标：解决插件前端展示偏袖珍、信息层级拥挤、新注册免费权益表达不足的问题，形成简洁、大方、可实施的界面重构方案。免费权益由后台统一配置，并通过自动充值进入业务充值与消费链路。

---

## 一、背景与问题诊断

### 1.1 当前状态

`apps/browser-extension` 已采用 WXT + React + Tailwind CSS 实现多入口浏览器插件，主要前端入口包括：

| 入口 | 当前职责 | 主要文件 |
|------|----------|----------|
| Popup | 登录、兑换码、文件/文件夹选择、转换参数、最近任务 | `src/entrypoints/popup/PopupApp.tsx` |
| Side Panel | 任务、套餐、反馈、账户工作台 | `src/entrypoints/sidepanel/SidePanelApp.tsx` |
| Options | API、主题、默认转换、域名权限等设置 | `src/entrypoints/options/OptionsApp.tsx` |

现有 Popup 使用 `max-w-popup = 420px`，最小高度约 400px，且认证、兑换、文件选择、转换模式、最近任务、设置入口全部压在同一个小窗口中。虽然功能完整，但用户感知更接近“工程工具窗”，而不是清晰可信的商业化转换产品入口。

### 1.2 主要问题

| 问题 | 表现 | 对用户影响 |
|------|------|------------|
| 展示过于袖珍 | Popup 宽度和高度偏小，文字多为 `text-xs` / `text-[11px]` | 新用户难以快速理解插件价值，操作显得局促 |
| 信息层级拥挤 | 登录、兑换、转换、任务列表混排 | 首次使用路径不够明确 |
| 免费权益不突出 | 新用户免费赠送权益缺少统一配置、充值来源和首屏承诺 | 注册动机弱，商业化转化链路不清晰 |
| 工作台定位不够鲜明 | Side Panel 已有任务/套餐/反馈/账户，但缺少账户概览首屏 | 已登录用户不能一眼看到额度、到期和下一步动作 |
| 视觉商业感不足 | 卡片较多、嵌套感明显，图标和按钮尺寸偏工具化 | 产品可信度和专业感不足 |

## 二、产品定位与设计原则

### 2.1 入口定位

| 入口 | 新定位 | 设计原则 |
|------|--------|----------|
| Popup | 快速转换入口 | 聚焦“权益提示、选择项目、开始转换、查看最近结果” |
| Side Panel | 主工作台 | 承载任务管理、套餐权益、账户状态、反馈支持 |
| Options | 设置中心 | 保持安静克制，只处理配置，不承载营销信息 |

### 2.2 设计关键词

1. **简洁**：减少并列小控件，突出主行动按钮。
2. **大方**：扩大 Popup 可视面积，提升字号、留白和模块间距。
3. **可信**：首屏明确展示免费权益、文件隐私、本地/云端转换差异。
4. **高效**：已登录用户能直接选择文件并转换，未登录用户能快速理解注册收益。
5. **一致**：Popup、Side Panel、Options 使用同一套颜色、间距、圆角和状态表达。

## 三、信息架构

### 3.1 用户状态

前端至少覆盖以下五种状态：

| 状态 | 展示重点 | 主要行动 |
|------|----------|----------|
| 未登录 | 免费套餐权益：根据后台配置展示注册赠送次数，默认 30 次 | 登录 / 注册 / 兑换 |
| 新注册免费套餐 | 剩余额度、有效期、首次转换引导 | 选择项目并转换 |
| 已登录且有额度 | 当前套餐、剩余次数、到期时间 | 继续转换 / 查看任务 |
| 额度不足 | 额度用尽原因、升级或充值入口 | 升级套餐 / 兑换码充值 |
| 套餐即将到期 | 到期提醒、剩余天数、续费入口 | 续费 / 升级 |

### 3.2 Popup 新结构

Popup 重构为三段式：

```text
┌──────────────────────────────────────────────┐
│ 顶部：品牌 + 账户状态 + 免费权益 / 剩余额度  │
├──────────────────────────────────────────────┤
│ 主体：选择 ZIP / 文件夹 + 主 TeX + 转换参数 │
├──────────────────────────────────────────────┤
│ 底部：转换进度 / 最近任务 / 工作台入口       │
└──────────────────────────────────────────────┘
```

建议尺寸：

| 项 | 建议值 |
|----|--------|
| Popup 宽度 | 480px |
| Popup 最小高度 | 560px |
| Popup 最大高度 | 600px |
| 内容内边距 | 20px |
| 主按钮高度 | 44px |
| 正文字号 | 13-14px |
| 辅助字号 | 12px |

### 3.3 Side Panel 新结构

Side Panel 作为工作台，保留现有 tabs，但增加“账户概览”作为顶部稳定区域：

```text
┌────────────────────────────────────┐
│ 账户概览：头像、邮箱、套餐、剩余额度 │
│ 免费套餐 / 到期 / 升级 CTA          │
├────────────────────────────────────┤
│ Tabs：任务 / 套餐 / 反馈 / 账户     │
├────────────────────────────────────┤
│ 当前 Tab 内容                       │
└────────────────────────────────────┘
```

### 3.4 Options 新结构

Options 保持配置中心定位：

- 不展示免费套餐营销横幅。
- 不放注册转化文案。
- 仅在“关于”中保留版本、官网、匿名埋点导出等信息。
- 设置项保持分组清晰，不增加额外商业入口。

## 四、免费套餐与自动充值规则

### 4.1 产品规则

新注册用户是否获得免费套餐、赠送多少次、有效期多久，均由后台统一配置。默认策略为：开启新注册送充值，默认赠送 30 次云端转换额度。

| 规则项 | 内容 |
|--------|------|
| 是否赠送 | 后台配置开关控制，默认开启 |
| 默认免费次数 | 30 次云端转换 |
| 免费次数配置 | 后台可配置为任意运营套餐次数 |
| 有效期 | 后台配置，建议默认 3 个月 |
| 发放时机 | 新账户注册成功后由服务端自动创建一笔注册送充值 |
| 发放方式 | 通过充值业务产生 `recharge` / `grant` 记录，而不是直接改用户余额 |
| 使用范围 | 云端转换额度，本地 WASM 转换不消耗该额度 |
| 到期处理 | 到期后未使用次数失效，引导升级或充值 |
| 用尽处理 | 剩余次数为 0 时禁用云端转换主按钮，引导升级或兑换 |

核心要求：

- 免费套餐必须通过“自动充值”产生，保证注册赠送、兑换码充值、人工充值、付费购买共用同一条余额增加链路。
- 后续转换消费必须从统一余额/额度体系扣减，保证消费流水、对账、过期处理和客服追溯连续。
- 前端不得本地发放免费额度，也不得绕过充值记录直接显示虚拟赠送次数。

### 4.2 后台配置建议

后台建议提供注册送充值配置项：

```ts
{
  enabled: true,
  grant_type: "signup_bonus",
  default_quantity: 30,
  validity_days: 90,
  package_name: "新用户免费体验包",
  recharge_source: "system_signup"
}
```

配置说明：

| 字段 | 说明 |
|------|------|
| `enabled` | 是否为新注册用户自动赠送 |
| `grant_type` | 赠送类型，建议使用 `signup_bonus` |
| `default_quantity` | 默认赠送次数，默认 30 |
| `validity_days` | 有效期天数，建议默认 90 天 |
| `package_name` | 充值包展示名称 |
| `recharge_source` | 充值来源，用于审计、对账和客服排查 |

### 4.3 充值与消费连续性

新注册赠送必须复用现有充值链路，推荐流程：

```text
用户注册成功
→ 读取后台注册送充值配置
→ 若 enabled=true，创建系统充值记录
→ 增加用户可用次数与有效期
→ REFRESH_SESSION / FETCH_USAGE 返回最新 UsageSummary
→ 用户云端转换时从统一额度中扣减
```

业务记录建议至少包含：

| 字段 | 说明 |
|------|------|
| `recharge_id` | 自动充值流水 ID |
| `user_id` | 用户 ID |
| `source` | `system_signup` |
| `quantity` | 本次赠送次数，默认 30 |
| `date_valid_until` | 本次赠送有效期 |
| `created_at` | 自动充值创建时间 |
| `operator` | `system` |

### 4.4 前端展示规则

前端不本地发放免费额度，不写死真实计费状态；所有额度以服务端返回的 `UsageSummary` 为准。

服务端推荐在注册后返回或刷新出：

```ts
{
  plan_id: "free_trial",
  cloud_conversions_used: 0,
  cloud_conversions_limit: 30,
  count_balance: 30,
  date_valid_until: "自动充值有效期"
}
```

说明：

- `cloud_conversions_limit` 与 `count_balance` 的具体含义应以后端现有额度模型为准。
- 若赠送次数进入 `count_balance`，Popup 和 Side Panel 应展示总剩余次数，而不是只看月度套餐额度。
- 前端文案应使用服务端返回或配置接口提供的次数与有效期，不硬编码 30。
- 30 次只是默认配置值，不是产品不可变常量。

### 4.5 文案建议

| 场景 | 中文文案 |
|------|----------|
| 未登录首屏 | 新用户注册可获免费云端转换额度，默认 30 次 |
| 后台关闭赠送 | 注册后可使用本地转换，也可购买或兑换云端额度 |
| 新注册成功 | 免费体验额度已到账：{quantity} 次，有效期至 {date} |
| 已登录有额度 | 剩余 {remaining} 次，套餐有效期至 {date} |
| 即将到期 | 套餐将在 {days} 天后到期，请及时续费 |
| 额度用尽 | 免费额度已用完，可升级套餐或使用兑换码充值 |
| 本地模式提示 | 本地转换不消耗云端额度，文件不会离开浏览器 |
| 云端模式提示 | 云端转换适合复杂模板、大项目和高质量输出 |

## 五、UI 重构方案

### 5.1 视觉方向

| 项 | 方案 |
|----|------|
| 主色 | 保留当前 `primary` 蓝色，但减少大面积渐变 |
| 背景 | 使用 `gray-50` / `white`，暗色模式使用 `gray-900` / `gray-800` |
| 卡片 | 仅用于独立内容块，避免卡片套卡片 |
| 圆角 | 控制在 8px 左右，商业工具界面保持克制 |
| 字号 | 主标题 16-18px，正文 13-14px，辅助 12px |
| 间距 | 模块间距 16-20px，控件内距 10-14px |
| 图标 | 后续代码实现建议使用 lucide 图标替代手写 SVG |

### 5.2 Popup 详细设计

#### 顶部权益区

未登录：

- 左侧展示 Tex2Doc 品牌与一句价值说明。
- 右侧保留语言切换。
- 主横幅展示后台配置的注册赠送权益，默认文案为“新用户注册可获 30 次免费云端转换额度”。
- 横幅下方提供“登录 / 注册”主按钮和“兑换码”次入口。

已登录：

- 展示用户邮箱缩略信息。
- 展示剩余额度：`剩余 {remaining} / {limit}`。
- 展示有效期：`有效期至 {date}`。
- 若即将到期，显示轻量 warning banner。

#### 文件选择区

将 ZIP 与文件夹选择合并为一个大选择区域：

- 主区域文案：`选择 LaTeX ZIP 或项目文件夹`
- 支持提示：`支持 .zip，也可直接选择文件夹`
- 已选择后展示：文件名 / 文件数 / 大小 / 自动识别主 TeX
- 多个 `.tex` 文件时，用列表选择主文件，但列表高度限制在 120px 内

#### 转换参数区

模式选择使用双段式控件：

| 模式 | 说明 |
|------|------|
| 本地转换 | 隐私优先，不消耗额度 |
| 云端转换 | 复杂项目，高质量输出，消耗额度 |

云端模式下才显示 `profile` 与 `quality`，避免未使用选项干扰。

#### 转换反馈区

- 转换中：显示阶段名称、进度条、当前动作。
- 成功：显示“转换完成”与最近任务入口。
- 失败：显示错误摘要，提供“重试”和“查看诊断 / 反馈”入口。

#### 底部操作区

底部保留两个清晰入口：

- `打开工作台`
- `设置`

其中“工作台”优先打开 Side Panel；不支持 Side Panel 的浏览器退化为 Options 或任务页入口。

### 5.3 Side Panel 详细设计

#### 账户概览

顶部固定展示：

- 头像 / 邮箱
- 套餐名称
- 剩余次数与进度条
- 有效期
- 主 CTA：有额度时为“开始新转换”，额度不足时为“升级套餐”

#### 任务页

- 保留任务列表。
- 任务卡片提升信息密度但减少小字堆叠。
- 完成任务突出“下载 DOCX”按钮。
- 展开态展示质量报告摘要。

#### 套餐页

套餐页顶部增加免费套餐卡片：

| 内容 | 说明 |
|------|------|
| 免费体验额度 | 根据后台配置自动发放 |
| 默认 30 次 | 默认云端转换赠送次数，可后台调整 |
| 默认 90 天 | 默认有效期，可后台调整 |
| 本地转换 | 不消耗额度 |

付费套餐列表继续来自 `FETCH_PLANS`。免费体验额度为后台注册送充值配置产生的权益说明卡，不参与结账，也不由前端发放。

#### 反馈页

- 保留诊断导出和反馈提交。
- 文案强调“诊断包不包含源文件内容、token、邮箱”。

#### 账户页

- 展示邮箱、套餐、额度、有效期。
- 提供账单门户、退出登录。
- 若为免费套餐，展示“免费套餐权益说明”和升级入口。

### 5.4 Options 详细设计

Options 不做营销化改造，只做视觉一致性微调：

- 顶部标题保持简洁。
- Tab 区域不使用重卡片包裹。
- 设置表单使用更清晰的 2 列布局。
- 保存/重置按钮固定在内容底部右侧。
- 域名权限页强化浏览器授权状态说明。

## 六、实施拆解

### 6.1 第一阶段：文档与设计锁定

1. 在 `docs-zh/extension` 新增本方案文档。
2. 明确免费权益由后台配置，并通过服务端自动充值发放，前端只展示 `UsageSummary`。
3. 确定 Popup 尺寸目标：480px 宽、约 560-600px 高。
4. 明确 Side Panel 为主工作台，Options 不承载营销信息。

### 6.2 第二阶段：前端结构重排

建议修改范围：

| 模块 | 改造内容 |
|------|----------|
| `PopupApp` | 三段式布局、免费权益横幅、统一文件选择区、底部工作台入口 |
| `SidePanelApp` | 增加账户概览区，套餐页增加免费体验额度说明卡 |
| `OptionsApp` | 轻量视觉一致性调整 |
| `i18n/index.ts` | 增加免费体验额度、自动充值、到期、升级相关中英文文案 |
| `quota-store.ts` | 统一剩余额度、有效期、展示文本计算 |
| `tailwind.config.ts` / `tokens.ts` | 调整 Popup 宽高 token |

### 6.3 第三阶段：状态与文案统一

统一抽象以下展示逻辑：

| 逻辑 | 输出 |
|------|------|
| 剩余额度 | `remaining = max(0, limit - used) + count_balance` |
| 免费体验额度识别 | 优先使用服务端返回的充值/套餐来源；无来源时仅按 `UsageSummary` 展示余额 |
| 到期天数 | 根据 `date_valid_until` 计算 |
| 是否即将到期 | 距到期日小于等于 7 天 |
| 是否可云端转换 | 已登录且 `remaining > 0` |

### 6.4 第四阶段：浏览器兼容验证

| 浏览器 | 验证重点 |
|--------|----------|
| Chrome | Popup 480x600 显示、Side Panel 工作台 |
| Edge | Side Panel 权限与入口 |
| Firefox | 无 Side Panel 时入口退化 |
| Safari | Popup 高度和文件选择能力 |

## 七、接口与后端协作要求

### 7.1 注册后自动充值

服务端必须保证新注册用户创建成功后，按照后台配置决定是否自动充值。若配置开启，则必须创建真实充值流水，并在 `REFRESH_SESSION` / `FETCH_USAGE` 中返回一致结果。

后台配置默认值：

| 配置项 | 默认值 |
|--------|--------|
| 是否开启注册送充值 | 开启 |
| 默认赠送次数 | 30 |
| 默认有效期 | 90 天 |
| 充值来源 | `system_signup` |

自动充值记录与用量返回建议字段：

| 字段 | 要求 |
|------|------|
| `recharge_id` | 必须存在，方便追踪注册送充值流水 |
| `recharge_source` | 推荐为 `system_signup` |
| `quantity` | 后台配置的赠送次数，默认 30 |
| `date_valid_until` | 自动充值有效期 |
| `plan_id` | 可推荐为 `free_trial`，但不作为唯一判断依据 |
| `cloud_conversions_used` | 新用户初始为 0 |
| `count_balance` / `cloud_conversions_limit` | 必须能让前端计算总剩余次数 |
| `period_start` / `period_end` | 与免费体验或当前套餐周期一致 |

### 7.2 前端不承担计费裁决

前端只做：

- 展示额度。
- 禁用明显不可用的云端转换按钮。
- 引导升级、续费、兑换。
- 在云端转换失败且服务端返回 quota 错误时展示错误提示。

前端不做：

- 本地发放免费额度。
- 本地重置用户套餐。
- 本地绕过额度调用云端转换。

## 九、风险与应对

| 风险 | 等级 | 应对 |
|------|------|------|
| Popup 变大后部分浏览器显示高度受限 | 中 | 控制最大高度 600px，内容区滚动，底部操作固定 |
| 免费权益展示与服务端真实额度不一致 | 高 | 前端只消费 `UsageSummary` 和服务端配置结果，注册赠送由自动充值链路保证 |
| 过度营销影响工具效率 | 中 | Popup 只在未登录或免费套餐期展示权益，Options 不放营销内容 |
| Side Panel 不被 Firefox/Safari 支持 | 中 | 提供 Popup / Options 退化入口 |
| 文案新增导致中英文不一致 | 低 | 所有新文案集中写入 `i18n/index.ts` |
| 卡片和按钮变大后信息密度下降 | 低 | 最近任务只显示 2-3 条，详细信息放 Side Panel |

## 十、结论

本次重构不改变插件核心转换能力，而是把 Tex2Doc 浏览器插件从“功能已经具备的小工具”升级为“新用户容易理解、已登录用户高效操作、商业权益表达清晰”的产品化前端。

最关键的产品承诺是：

```text
新注册用户是否获得免费云端转换额度、获得多少次、有效期多久，均由后台统一配置；默认赠送 30 次。
```

最关键的实现边界是：

```text
免费权益由服务端通过自动充值发放和裁决，前端只负责展示、引导和状态刷新。
```

最关键的界面方向是：

```text
Popup 做大且聚焦，Side Panel 做完整工作台，Options 保持设置中心。
```
