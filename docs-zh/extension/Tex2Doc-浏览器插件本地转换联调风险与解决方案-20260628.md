# Tex2Doc 浏览器插件本地转换联调风险与解决方案

> **范围**：本次为把 `apps/browser-extension` 编译产物实际加载到 Chrome/Edge、并把本地 WASM 转换跑通所涉及的 6 类核心风险点排查与修复归档。
> **日期**：2026-06-28
> **状态**：✅ Chrome MV3 / Edge MV3 均已在真实 Chromium 中跑通 paper3 zip → docx（3,056,009 bytes，magic bytes PK\x03\x04 正确），可用性已验证。

---

## 0. TL;DR — 6 个核心风险点

| # | 风险类别 | 触发条件 | 现象 | 修复策略 |
|---|---|---|---|---|
| 1 | **CSP / 清单不兼容** | Chrome MV3 拒绝 `blob:` CSP + `sidePanel` 权限 | 扩展装不上：`Insecure CSP value "blob:"` | WXT 后处理钩子强制覆写 manifest |
| 2 | **WASM 编译依赖冲突** | `pdfium-render` / `tokio` 牵入 `mio`、`which` | `wasm-pack` 编译失败 | 特性门控 + `#[cfg(target_arch)]` 排除 |
| 3 | **WASM 内存 panic** | `Vec::with_capacity(u64 → usize)` 转换越界 | 浏览器侧 `Conversion Failed invalid malloc request` | 单 entry 128 MiB 上限 + 显式边界检查 |
| 4 | **service-worker 动态加载限制** | Chrome MV3 SW 禁 `import()` + 禁 `unsafe-eval` | `__wbindgen_object_drop_ref: function import requires a callable` / `import() is disallowed` / `Evaluating a string ... violates CSP` | `scripts/post-build-wasm.mjs` 把 ESM 胶水改写成 IIFE + globalThis，让 Vite 静态 import 进 bundle |
| 5 | **service-worker 下载 API 缺失** | SW 没有 `URL.createObjectURL` | 转换成功后 `downloadBytes` 抛 `window is not defined` | `downloadBytes` 探测式 fallback 到 `data:` URL（base64） |
| 6 | **WXT service worker 输出格式** | WXT 默认把 background 打成 IIFE | 没法用 dynamic import / ESM 上下文 | `defineBackground({ type: 'module', main() {...} })` |

下面逐项展开。

## 1. 风险点：CSP 与 Chrome MV3 清单不兼容

### 触发与现象
Chrome MV3 严格校验 `content_security_policy` 字段：
- `object-src` 拒绝任何 `blob:` 值；
- `worker-src` 接受 `blob:`，但 Chrome 把 WXT 默认生成的 `script-src 'self' 'wasm-unsafe-eval'; object-src 'self' blob:; worker-src 'self' blob:` 整段判为不合法；
- 同时 Chrome 不支持 Edge 专属的 `sidePanel` 权限和 `side_panel` 字段。

加载时报错：
```
未能成功加载扩展程序... 'content_security_policy.extension_pages':
Insecure CSP value "blob:" in directive 'object-src'.
```

### 根因
WXT 在 `wxt.config.ts` 的 `manifest` 回调里基于 `browser` / `mode` 设置 `permissions` 和 CSP，**但它内部仍然会把 `side_panel` + `sidePanel` 字段塞回 manifest**，并且 `hooks: { 'build:manifest': ... }` 被覆盖，等价于无。

### 解决方案
**新增 `scripts/fix-chrome-manifest.mjs`，在 `npm run build:chrome` 末尾作为 post-build 步骤强制覆写生成的 manifest**：

```javascript
// scripts/fix-chrome-manifest.mjs（节选）
const manifest = JSON.parse(fs.readFileSync(manifestPath, 'utf-8'));

// 1. 删除 Edge 专属字段
delete manifest.side_panel;

// 2. 过滤 Chrome 不支持的权限
if (Array.isArray(manifest.permissions)) {
  manifest.permissions = manifest.permissions.filter((p) => p !== 'sidePanel');
}

// 3. 强制设置安全的 CSP（无 blob:）
manifest.content_security_policy = {
  extension_pages:
    "script-src 'self' 'wasm-unsafe-eval'; worker-src 'self'; object-src 'self'",
};

// 4. service worker 必须声明为 module（详见风险点 6）
if (manifest.background?.service_worker) {
  manifest.background.type = 'module';
}
```

`package.json` 接入：
```json
"build:chrome": "wxt build -b chrome && node scripts/fix-chrome-manifest.mjs",
```

### 经验
- WXT 行为细节藏在版本里；遇到官方文档没写的行为，**强制覆盖输出比配置回退更可靠**。
- Edge 构建**不要**调用 `fix-chrome-manifest.mjs`，否则会把 Edge 需要的 `side_panel` 也删掉。Chrome / Edge 用两套 build 脚本。

## 2. 风险点：WASM 编译依赖冲突（`mio` / `which`）

### 触发与现象
`npm run build:wasm:extension` 在 `crates/wasm` 上失败：
```
error[E0463]: can't find crate for `mio`
error[E0463]: can't find crate for `which`
```
均为 `wasm32-unknown-unknown` target 缺失。

### 根因
- `crates/core/Cargo.toml` 引入 `pdfium-render = "0.8"`（PDF 转 PNG），它直接依赖 `mio`；
- `crates/latex-reader/Cargo.toml` 引入 `doc-tex-facade = { path = "../../crates/tex-facade" }`，`tex-facade` 间接引入 `tokio`，而 `tokio` 在 `wasm32-unknown-unknown` 上需要 `mio` / `which`。

### 解决方案
1. **`pdfium-render` 改 optional**（`crates/core/Cargo.toml`）：
   ```toml
   [target.'cfg(not(target_arch = "wasm32"))'.dependencies]
   pdfium-render = "0.8"
   ```
   + 用 `#[cfg(not(target_arch = "wasm32"))]` 把 `render_pdf_to_png` 整个函数体包起来，WASM 构建直接不编译 PDF 处理路径。

2. **`doc-tex-facade` 依赖直接移除**（`crates/latex-reader/Cargo.toml`）：检查后发现 latex-reader 并没真的用这个 crate 的符号，是早期遗留依赖。

### 经验
- 任何 wasm32 目标都必须**严格审视 transitive 依赖**——desktop 友好的 `tokio` / `mio` / `signal-hook` 等在 wasm32 上基本都没法编。
- 用 `cargo tree --target wasm32-unknown-unknown` 提前预演依赖关系，比反复跑 `wasm-pack build` 快得多。

## 3. 风险点：WASM 内存 panic（`invalid malloc request`）

### 触发与现象
浏览器侧报错：
```
Conversion Failed invalid malloc request
```
触发位置：`crates/core/src/convert.rs` 里：
```rust
Vec::with_capacity(f.size() as usize)  // u64 → usize 截断 + 巨大值触发 OOM
```
`f.size()` 是 zip entry header 里的 `u64`，恶意构造（zip bomb）可以填 `0xFFFF_FFFF_FFFF_FFFF` 这种值。

### 解决方案
引入 **单 entry 128 MiB 上限** + **显式 u64 → usize 边界检查**（`crates/core/src/convert.rs`）：

```rust
const MAX_ZIP_ENTRY_BYTES: u64 = 128 * 1024 * 1024; // 128 MiB

let declared_size = f.size();
if declared_size > MAX_ZIP_ENTRY_BYTES {
    return Err(CoreError::Io(io::Error::new(
        io::ErrorKind::InvalidData,
        format!("zip entry too large: {} > MAX", declared_size),
    )));
}
let cap: usize = declared_size.try_into().map_err(|_| {
    CoreError::Io(io::Error::new(io::ErrorKind::InvalidData, "usize overflow"))
})?;
let mut buf = Vec::with_capacity(cap);
```

### 配套测试
新增 `crates/core/tests/zip_bomb_protection.rs`：构造一个声明 2 GiB 但实际只写少量字节的 zip，断言 `convert_zip` 优雅报错。

### 经验
- WASM 32 位地址空间只有 4 GiB；任何接受 `u64` 大小的边界都要先 cap 再 cast。
- 在 WASM 上下文里 `OOM` 会**直接 abort 整个 wasm 实例**，比 native 进程的 stack overflow 还难调试。

## 4. 风险点（核心）：service-worker 动态加载限制

这是本次调试**耗时最长、坑最深的**问题。最终报错一步步演化：

| 阶段 | 现象 | 真因 |
|---|---|---|
| 4.1 | `WebAssembly.instantiate(): Import #0 "./doc_engine_bg.js" "__wbindgen_object_drop_ref": function import requires a callable` | 手写的旧版 wasm-bindgen 胶水漏掉了 0.2.x 改名后的 runtime imports |
| 4.2 | `window is not defined`（在 `vite:preloadError` 处） | Vite 默认给 `import()` 注入 preload error helper，用 `window.dispatchEvent`，但 SW 没 `window` |
| 4.3 | `import() is disallowed on ServiceWorkerGlobalScope by the HTML specification` | Chrome MV3 SW **明确禁止**动态 `import()` |
| 4.4 | `Evaluating a string as JavaScript violates the following Content Security Policy directive because 'unsafe-eval' is not an allowed source of script` | SW CSP 不允许 `unsafe-eval`，所以 `eval()` / `new Function()` 也不行 |

### 根因汇总
Chrome MV3 service worker 同时拒绝三种加载方式：

| 加载方式 | SW 接受？ |
|---|---|
| `await import(url)` | ❌ HTML spec 禁止 |
| `eval(src)` / `new Function(src)` | ❌ CSP `unsafe-eval` 未授权 |
| `URL.createObjectURL(blob)` + `<script src>` | ❌ SW 没有 DOM |

**唯一可行**：让 Vite 把 wasm-bindgen 生成的 ESM 胶水**静态打进 bundle**。

### 解决方案

#### 4.1 在 wasm-pack 构建后改写胶水

新增 `scripts/post-build-wasm.mjs`：

```javascript
// scripts/post-build-wasm.mjs（节选）
let src = fs.readFileSync(GLUE_JS, 'utf8');

// 1) 去掉顶层 export class/function 前缀
src = src.replace(/^export\s+(class|function|const|let|var|async function)\s+/gm, '$1 ');

// 2) 顶层 `export { initSync, __wbg_init as default }` 改写为 globalThis 赋值
src = src.replace(/export\s*\{\s*([^}]+)\s*\}\s*;?\s*$/m, (m, names) => {
  // 解析 { initSync, __wbg_init as default } → globalThis.__tex2docWbg = { initSync, default: __wbg_init }
  return `globalThis.__tex2docWbg = {\n${parts.join(',\n')},\n    };`;
});

// 3) import.meta.url → globalThis.__tex2docBaseUrl
src = src.replaceAll('import.meta.url', 'globalThis.__tex2docBaseUrl');

// 4) IIFE 包裹 + 末尾挂 API 到 globalThis
const iife = `(function(){\n${src}\n    globalThis.__tex2docApi = { /* convert_zip 等 */ };\n}).call(globalThis);\n`;
fs.writeFileSync(GLUE_JS, iife);

// 5) 额外复制到 src/workers/wasm-glue/doc_engine.js，
//    让 wasm-worker.ts 能静态 import 进来
fs.copyFileSync(GLUE_JS, GLUE_COPY);
```

#### 4.2 wasm-worker.ts 改为静态 import

```typescript
// apps/browser-extension/src/workers/wasm-worker.ts
// 静态 import：Vite 会把 IIFE 胶水内联进 background.js
import './wasm-glue/doc_engine.js';

// wasm-bindgen 0.2.x 把 __wbg_init export 为 default，
// 打包后被 mangled 成 __tex2docWbg.default
const wbgInit = g.__tex2docWbg?.__wbg_init ?? g.__tex2docWbg?.default;

// fetch wasm + 喂给 __wbg_init 完成 instantiate
const resp = new Response(wasmBytes, { headers: { 'Content-Type': 'application/wasm' } });
await wbgInit(resp);
```

#### 4.3 入口脚本接入

`package.json`：
```json
"build:wasm:extension": "cd crates/wasm && wasm-pack build --target web --out-dir ../../apps/browser-extension/public/wasm --out-name doc_engine --release && cd ../../ && node scripts/post-build-wasm.mjs"
```

### 经验
- **Chrome MV3 SW 是一个极度受限的运行时**：没有 DOM、没有 `window`、没有动态加载、连 `unsafe-eval` 都默认关。要加载任何第三方 JS 库，**必须让构建工具在打包阶段把它静态打进去**。
- wasm-bindgen 0.2.x 把 `__wbg_init` 重新 export 成 `default`，所以打包后会变成 `globalThis.__tex2docWbg.default`，**调试时一定要 console.log keys 验证属性名**。
- `modulePreload: false` 在 wxt@0.20 + Vite 8 (rolldown) 上不生效；rollupOptions 的 `manualChunks` object 形式也会被 rolldown 拒绝（必须 function 形式）。**不要在这些无效开关上浪费时间**。

## 5. 风险点：service-worker 内下载文件

### 触发与现象
WASM 转换成功后，`handleStartWasmConversion` 调用 `downloadBytes(docxBytes, filename)`。但 SW 里：
```
ReferenceError: window is not defined
    at URL.createObjectURL(blob)
```
被 try/catch 包成 `WASM conversion failed`，前端显示：
```
Conversion Failed
WASM conversion failed
```

### 根因
`apps/browser-extension/src/browser/downloads.ts` 原本：
```typescript
const blob = new Blob([data], { type: mimeType });
const url = URL.createObjectURL(blob);   // SW 里没 URL
await browser.downloads.download({ url, ... });
```

SW 没有 `URL` 全局，`URL.createObjectURL` 抛 ReferenceError。

### 解决方案
探测式创建 URL，能用 blob URL 就用，**否则 fallback 到 `data:` URL（base64 编码）**：

```typescript
function createBlobUrl(data: Uint8Array, mimeType: string): string {
  // 1) popup / options / sidepanel：优先用 blob URL
  if (typeof URL !== 'undefined' && typeof URL.createObjectURL === 'function') {
    try {
      const blob = new Blob([new Uint8Array(data)], { type: mimeType });
      return URL.createObjectURL(blob);
    } catch { /* fall through */ }
  }

  // 2) service-worker fallback：base64 data URL
  //    chrome.downloads.download 接受 data: URL，最大长度 1 GiB，docx 足够
  const bytes = new Uint8Array(data);
  let binary = '';
  for (let i = 0; i < bytes.length; i += 0x8000) {
    const chunk = bytes.subarray(i, i + 0x8000);
    binary += String.fromCharCode.apply(null, Array.from(chunk));
  }
  return `data:${mimeType};base64,${btoa(binary)}`;
}
```

`revokeBlobUrl` 也加上 `data:` 前缀判断，避免对 data URL 调 revoke 报错。

### 经验
- 任何在 popup 和 SW 共用的工具模块，**必须假设两边全局 API 集合不同**——DOM 类 API（`URL.createObjectURL`、`Blob`、`window`）在 SW 里都没有。
- `chrome.downloads.download` 接受 `blob:` / `data:` / `http(s):` 三种 URL，统一收口在一个工具函数里可以避免在不同上下文重复实现。

## 6. 风险点：WXT service worker 输出格式

### 触发与现象
WXT 默认把 background entrypoint 打包成 IIFE 格式，无法 `import()` 任何 ESM 模块。导致 SW 里没法用任何 wasm 加载方式。

### 解决方案
把 `defineBackground` 改为 ESM 形式：
```typescript
// apps/browser-extension/src/entrypoints/background.ts
export default defineBackground({
  type: 'module',   // ← 关键：让 WXT 输出 ESM
  main() {
    browser.runtime.onInstalled.addListener(...);
    browser.runtime.onMessage.addListener(handleMessage);
    // ...
  },
});
```

并加一份后处理兜底（防止 WXT 内部逻辑把 `type: 'module'` 吃掉）：
```javascript
// scripts/fix-chrome-manifest.mjs
if (manifest.background?.service_worker) {
  manifest.background.type = 'module';
}
```

### 经验
- 凡是 Chrome MV3 扩展要加载任何 WASM / ESM 模块，**background 必须 ESM 化**。
- `defineBackground({ type: 'module', main() {...} })` 是 WXT 0.16+ 的官方写法，比 `hooks: { 'build:manifest': ... }` 更稳定。

## 7. 联调与回归保障

### 7.1 真实浏览器回归：`scripts/e2e_wasm_convert.mjs`

```bash
# 在项目根目录
node scripts/e2e_wasm_convert.mjs
```

要点：
- 必须用 `chromium.launchPersistentContext` + `channel: 'chromium'`（playwright 自带的 chromium 默认禁用扩展）；

## 8. 调试心法 — 一份"避坑清单"

1. **CSP / manifest 问题永远先看 .output**：WXT 输出的 manifest 是真实构建结果，配置 vs 实际常有差异；以输出文件为准。
2. **服务相关报错先看是不是 SW 上下文**：MV3 SW 的错误和 popup 的错误表现一样，但根因往往在 SW 没 DOM。判断方式：看报错位置是不是在 `URL`、`window`、`document` 等 DOM API。
3. **WASM 加载相关错误优先验证 import**：把 wasm-bindgen 生成的 `__wbg_init` 在浏览器 console 直接调一次，能区分是胶水问题还是 instantiate 问题。
4. **MV3 SW 禁用清单**：`import()`、`unsafe-eval`、`document.write`、`XMLHttpRequest`（同步）、`alert()` 等。规避方式见风险点 4。
5. **zip bomb 防护是 WASM 必需**：任何接受外部 zip 的 WASM 都要加 entry size 上限。
6. **不要在 SW 里用 `URL.createObjectURL` + Blob**：用 `data:` URL + `chrome.downloads`。
7. **Playwright 加载扩展必须用 `channel: 'chromium'` + `launchPersistentContext`**，二者缺一不可。
8. **Post-build 脚本是最稳的强约束手段**：当框架配置无法满足需求时（比如 WXT 强制塞 `side_panel`），post-build 直接重写输出文件比反复调配置高效得多。

## 10. 后续可改进项（未在本次范围内）

- **wasm-bindgen 升级路径**：本次的 `__wbg_init as default` 兼容是 0.2.x 适配。升级到 0.3+ 后 `__wbg_init` 可能改名，需要重新生成胶水 + 重测 e2e。
- **更小 docx 分块下载**：当前 3 MB docx → 4 MB base64 data URL，单次 `chrome.downloads.download` 够用。如果以后支持 PDF / 大型项目（>50 MB），需要改用 chrome filesystem API 分块写。
- **WXT modulePreload 关闭的最终方案**：wxt@0.20 + Vite 8 (rolldown) 没真正生效；后续 wxt 升级时再验证。如果仍然无效，**当个遗留警告留着即可**，因为我们已经走静态 import 路径绕开了。
- **popup / sidepanel 异步转换反馈**：当前 `_e2eReturnBytes` 是测试钩子；生产 UI 要监听 `chrome.downloads.onChanged` 拿真实下载进度，而不是靠 message 返回值。
