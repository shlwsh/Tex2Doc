# Tex2Doc React 商业化推广界面级重构设计实现方案

> 日期：2026-06-29  
> 适用项目：`apps/react-web`  
> 目标阶段：从初版 Web 工作台升级为可面向商业化推广、销售演示、用户自助转化和运营交付的产品界面。  
> 参考方法：`.claude/skills/commercial-ui-design/SKILL.md`，其核心流程为先审计现有界面，再定义设计 token、组件体系、国际化、响应式状态和验证清单。

## 1. 总体结论

当前 `apps/react-web` 已经覆盖 Tex2Doc Web 用户端和管理端的主要业务闭环：快捷助手、会员中心、充值兑换、云端转换、反馈、兑换码运营、发布管理、审计和自动化管理。它适合作为功能原型与接口验证基线，但不宜直接用于商业化推广。

建议进行一次界面级重构，而不是重写后端或业务 API。重构重点是：

1. 将 `App.tsx` 中集中实现的页面拆分为可维护的产品模块。
2. 建立稳定的商业化设计系统：颜色、字体、间距、状态、表格、表单、反馈、空态、加载态。
3. 把“能操作”升级为“可信赖”：清晰的转换路径、额度说明、支付/兑换结果、管理员风险状态和审计可追溯。
4. 统一中文与英文文案，面向销售演示、海外学术用户和后台运营人员。
5. 保留当前 API SDK、Zustand 会话、WASM 懒加载和额度扣减顺序，只重构展示层与交互组织。

## 2. 现状审计

### 2.1 已具备能力

| 范围 | 当前能力 |
|---|---|
| 路由 | `/react`、`/app-react/*`、`/admin-react/*` 已分离产品入口、用户端、管理端。 |
| 用户端 | 快捷助手、账号、充值、转换、转换记录、充值记录、反馈、关于。 |
| 管理端 | 仪表盘、账号、兑换码生成、批次、库存、反馈、发布、审计、自动化。 |
| 会话 | `userSession`、`quickSession`、`adminSession` 隔离存储。 |
| 主题 | light/dark 基础变量和持久化偏好。 |
| 国际化 | 已有 `locale` 偏好，兑换码批次页面有局部 `batchText`。 |
| API | `Tex2DocApi` 已按用户端和管理端能力封装，适合作为重构边界。 |

### 2.2 商业化短板

| 问题 | 影响 | 改造方向 |
|---|---|---|
| 页面和组件集中在 `App.tsx` | 后续设计迭代、测试、多人协作成本高。 | 按 feature 拆分页面，按 design system 拆分基础组件。 |
| 中英文文案混杂 | 商业演示显得粗糙，英文用户体验断裂。 | 建立 `src/i18n/messages.ts`，全量使用稳定 key。 |
| 首页偏技术入口 | 对首次访问用户缺少“我该做什么”的业务引导。 | 改为产品工作台入口：立即转换、兑换激活、会员登录、管理入口。 |
| 快捷助手像调试面板 | 兑换码、API Base URL、日志并列出现，降低信任感。 | 改为 3 步转换流程，技术项收进高级设置。 |
| 管理端缺少信息层级 | 数据表和表单堆叠，风险、异常、待办不突出。 | 仪表盘聚合经营指标、风险任务、近期活动。 |
| 状态体系不完整 | loading、empty、error、disabled、permission denied 表现不一致。 | 抽象 `StateView`、`InlineNotice`、`Toast`、`PermissionBlock`。 |
| 表格能力不足 | 搜索、筛选、批量、导出、分页和详情侧栏体验不稳定。 | 统一 `DataTable` 与 `ResourceList` 模式。 |
| 移动端只是折叠 | 管理端导航项过多时移动端 segmented 难用。 | 用户端优先移动端，管理端定义平板以下只保留关键操作。 |

## 3. 产品定位

### 3.1 面向对象

| 用户 | 核心诉求 | 界面关键词 |
|---|---|---|
| 学术作者 / 研究生 | 快速把 LaTeX 项目转成可交付 DOCX。 | 简洁、可信、低学习成本。 |
| 付费会员 | 查看额度、发起云端转换、下载历史、充值续费。 | 清晰权益、进度可见、记录可追溯。 |
| 渠道销售 / 运营 | 生成和管理兑换码，处理客户反馈。 | 高密度、可筛选、可导出、少误操作。 |
| 产品管理员 | 看经营数据、发布版本、审计变更、处理自动化请求。 | 状态优先、风险优先、审计完整。 |

### 3.2 首屏承诺

商业化版本首屏不应先展示技术实现，而应明确传达：

1. Tex2Doc 是 LaTeX 到 DOCX 的产品化转换工具。
2. 用户可以立刻使用兑换码或账号开始转换。
3. 管理端是运营工作台，不与普通用户路径混在一起。
4. 任何额度消耗、云端任务、失败原因、下载结果都可见、可追溯。

## 5. 设计系统

### 5.1 设计原则

1. SaaS 工具界面优先：安静、清晰、对齐强、可扫描。
2. 商业化推广不靠大面积装饰，而靠可信的流程、状态和结果。
3. 卡片只用于信息单元、表单块、指标块和详情面板，不把整个页面套成层层卡片。
4. 所有可见状态都有明确反馈：正常、加载、空、错误、禁用、无权限。
5. 不再出现混杂文案：页面文案统一由 i18n key 驱动。

### 5.2 Token 建议

| Token | Light | Dark | 用途 |
|---|---|---|---|
| `--color-bg` | `#f6f7f9` | `#101316` | 页面背景。 |
| `--color-surface` | `#ffffff` | `#171b20` | 主内容面。 |
| `--color-surface-muted` | `#eef1f4` | `#20262d` | 次级面、表头、hover。 |
| `--color-text` | `#17202a` | `#eef2f6` | 主文字。 |
| `--color-muted` | `#667085` | `#aab3bf` | 辅助文字。 |
| `--color-line` | `#d9e1ea` | `#34404d` | 边框和分割线。 |
| `--color-primary` | `#1769d2` | `#6aa8ff` | 主操作、链接、选中态。 |
| `--color-success` | `#0f8a61` | `#4fd1a1` | 成功、已完成、额度有效。 |
| `--color-warning` | `#a86600` | `#f4b44d` | 待处理、额度将尽、风险提示。 |
| `--color-danger` | `#c24132` | `#ff8f7f` | 失败、危险操作。 |

### 5.3 尺寸与排版

| 项 | 建议 |
|---|---|
| 字体 | 保持 system-ui / Inter；中文走系统黑体，不引入额外字体包。 |
| 基准字号 | body 14px，表格 13px，页面标题 20-24px，产品入口标题 36-48px。 |
| 间距 | 4px 基准：4、8、12、16、20、24、32。 |
| 圆角 | 6px 为默认；面板和表格 8px；不要使用过度圆角。 |
| 阴影 | 仅用于浮层、弹窗、下拉和 sticky 区域；常规面板以边框为主。 |
| 动效 | 120-180ms，用于 hover、展开、toast、dialog；不做装饰性长动画。 |

### 5.4 状态语义

| 状态 | 展示 |
|---|---|
| `idle` | 文案提示下一步，不显示空白大区域。 |
| `loading` | 按钮 spinner + 区块 skeleton，避免整页闪烁。 |
| `ready` | 显示数据、更新时间和刷新动作。 |
| `empty` | 说明为什么为空，并给主操作。 |
| `error` | 错误摘要、错误详情、重试动作，技术详情折叠。 |
| `disabled` | 明确前置条件，例如“请先激活兑换码”。 |
| `permissionDenied` | 说明当前账号角色不满足，并提供退出或切换账号。 |

## 7. 用户端界面方案

### 7.1 产品入口 `/react`

首屏改为“产品入口 + 快速开始”：

1. 左侧是产品名称、简短价值说明、立即转换按钮。
2. 右侧是可操作的轻量转换入口：兑换码输入、ZIP 上传占位、开始按钮。
3. 管理端入口放在顶部或页脚的小入口，不与普通用户 CTA 抢焦点。
4. 展示 3 个可信信号：本地转换不上传、云端队列可追踪、额度扣减可查看。

首屏不展示 `Local WASM lazy`、`Cloud Queue 120 polls` 这类工程词，改为面向用户的结果语言。

### 7.2 快捷助手

当前快捷助手建议重构为 3 步：

| 步骤 | 页面行为 |
|---|---|
| 1. 激活权益 | 输入兑换码，显示权益类型、有效期、剩余额度。API Base URL 放入高级设置。 |
| 2. 上传项目 | 拖拽 ZIP，自动识别主 TeX，展示文件大小、文件数、检测到的候选入口。 |
| 3. 转换与下载 | 选择本地/云端、profile、quality，展示转换进度、扣减结果和下载按钮。 |

日志保留，但作为“技术详情”折叠面板。普通用户默认只看到进度、结果和可操作下一步。

### 7.3 会员中心

会员中心默认页应从“账号”改为“工作台”：

| 模块 | 内容 |
|---|---|
| 权益摘要 | 当前套餐、云端额度、按次余额、有效期、即将过期提醒。 |
| 快速转换 | 上传 ZIP、主 TeX 自动识别、转换选项。 |
| 最近任务 | 最近 5 条转换记录，状态、耗时、下载、失败原因。 |
| 充值入口 | 兑换码输入、套餐购买入口、充值记录。 |

转换记录页面需要强化“重新下载”“下载日志”“查看失败原因”“复制任务 ID”。

### 7.4 反馈

用户反馈界面改为会话式支持中心：

1. 左侧反馈列表，右侧消息详情。
2. 新建反馈时支持关联转换任务。
3. 状态清晰显示：待处理、处理中、已解决、已关闭。
4. 错误反馈可附带转换 job id、错误码、客户端版本。

## 8. 管理端界面方案

### 8.1 管理端仪表盘

仪表盘承担“今天要看什么、先处理什么”的作用：

| 区块 | 内容 |
|---|---|
| 经营指标 | 新增用户、转换任务、成功率、兑换码消耗、活跃权益。 |
| 待处理 | 高优先级反馈、失败转换、待审批自动化请求、发布异常。 |
| 风险概览 | 高风险自动化、最近回滚、管理员失败登录或权限异常。 |
| 趋势 | 近 7 日转换量、失败率、兑换码使用趋势。 |
| 最近活动 | 发布、兑换码批次、反馈状态变更、自动化审批。 |

### 8.2 兑换码运营

兑换码生成、批次、库存建议合并成“兑换码”分组：

| 页面 | 改造 |
|---|---|
| 批次 | 顶部筛选、批次表格、右侧详情抽屉、导出按钮、状态徽标。 |
| 库存 | 支持批次、套餐、状态、关键词筛选；批量入库、批量导出、批量复制。 |
| 生成 | 使用分步表单：套餐与数量、渠道与有效期、确认与生成结果。 |

危险动作，例如批量入库、导出全部、撤销批次，需要 `ConfirmDialog` 并显示影响数量。

### 8.3 客户支持

管理端反馈页面建议：

1. 使用 inbox 模式：左侧队列，右侧详情。
2. 队列支持状态、优先级、反馈类型、关联任务筛选。
3. 回复框支持公开回复和内部备注切换。
4. 处理完成后要求选择解决类型，便于后续分析。

### 8.4 发布与审计

发布管理需要从“表单 + 表格”升级为发布工作流：

| 步骤 | 内容 |
|---|---|
| 新建发布 | channel、platform、arch、version、下载地址、sha256。 |
| 校验 | URL 格式、sha256 长度、版本号重复、渠道策略。 |
| 发布 | 显示影响平台和回滚入口。 |
| 审计 | 每次发布、回滚、失败都可追踪 actor、时间和 details。 |

审计中心默认展示最新 100 条并支持按 action、actor、时间筛选。

### 8.5 自动化

自动化页面是高风险运营模块，应突出风险控制：

1. 请求列表中强展示 risk level、status、assignee、更新时间。
2. 高风险和 critical 请求不可直接 approve，界面必须说明原因。
3. 详情页按“摘要、差异、验证结果、事件流、操作区”组织。
4. Agent 列表显示在线状态、最近心跳、成功/失败计数、暂停/恢复动作。

## 9. 国际化方案

### 9.1 目标

至少完整支持 `zh-CN` 和 `en-US`。所有用户可见文案移入集中资源，避免出现当前 `Feedback management`、`Admin Gate`、`Profile` 等混杂状态。

### 9.2 Key 结构


### 9.3 落地规则

1. 页面不直接写固定展示文案，除非是 API 数据或用户文件名。
2. 枚举值统一走 formatter，例如 `formatStatus('in_progress')`。
3. 日期、数字、额度使用 locale 格式化。
4. 超长英文和中文均需在按钮、表格、徽标中验证不溢出。

## 10. 表单与校验

建议引入或继续使用 `zod` 做表单 schema。若后续加入 `react-hook-form`，优先改造复杂表单：兑换码生成、发布管理、反馈创建。

| 表单 | 校验 |
|---|---|
| 登录注册 | email、密码长度、管理员角色门禁错误。 |
| 兑换码激活 | 非空、去空格、错误码友好提示。 |
| ZIP 上传 | `.zip`、小于 10MB、能读取 ZIP、至少一个 `.tex`。 |
| 云端转换 | upload id、main tex、profile、quality。 |
| 兑换码批次 | quantity 范围、package id、expires at、channel。 |
| 发布 | semver、URL、sha256、channel/platform/arch。 |

## 11. 数据展示规范

### 11.1 表格

1. 默认提供加载态、空态、错误态。
2. 表头固定，长表格横向滚动但保持操作列可见。
3. ID 默认截断，点击可复制完整 ID。
4. 时间统一显示本地时间，同时 tooltip 提供 ISO 原始值。
5. 状态列使用语义 badge，不直接展示原始英文。
6. 支持筛选后的导出，导出按钮文案明确范围。

### 11.2 指标

指标卡必须说明时间范围，例如“今日”“近 7 日”“当前权益”。不要只显示孤立数字。

### 11.3 错误信息

错误分三层：

1. 用户摘要：例如“转换失败，未扣减额度”。
2. 可操作建议：例如“检查主 TeX 路径或下载日志”。
3. 技术详情：HTTP status、error_code、job id，默认折叠。

## 12. 响应式策略

| 断点 | 策略 |
|---|---|
| `< 640px` | 用户端单列；底部导航；表格转卡片列表；管理端复杂页提示桌面体验更佳。 |
| `640-1024px` | 双列变单列；详情抽屉全屏；顶部操作折叠进菜单。 |
| `> 1024px` | 管理端左侧导航 + 内容区；表格和详情抽屉并排。 |
| `> 1440px` | 内容最大宽度约 1440px，避免指标和表格过度拉伸。 |

## 13. 前端实现步骤

### Phase 0：界面基建

## 14. 验收标准

| 类别 | 标准 |
|---|---|
| 商业观感 | 首屏能让新用户 5 秒内理解产品用途并开始转换。 |
| 用户转化 | 兑换码激活、ZIP 上传、转换下载形成 3 步闭环。 |
| 运营效率 | 管理端可在仪表盘看到待办与风险，表格支持筛选、批量和导出。 |
| 国际化 | `zh-CN` 和 `en-US` 页面无混杂固定文案。 |
| 状态完整 | 主要页面都有 loading、empty、error、disabled、permission denied。 |
| 可维护性 | 页面不再集中在单个 `App.tsx`，功能模块和基础组件边界清晰。 |
| 兼容性 | 当前 API SDK、WASM 懒加载、额度扣减顺序不被破坏。 |
| 构建 | `npm run typecheck`、`npm run test`、`npm run build` 通过。 |

## 15. 风险与控制

| 风险 | 等级 | 控制 |
|---|---:|---|
| 视觉重构影响业务流程 | 高 | 保留 `Tex2DocApi` 方法签名，页面分步替换，每个功能先做 smoke test。 |
| 额度扣减顺序被误改 | 高 | 快捷助手仍保持 check -> convert -> consume -> download，测试覆盖成功和失败路径。 |
| i18n 改造范围大 | 中 | 先从导航、按钮、状态、核心用户端开始，再覆盖管理端长尾字段。 |
| 表格重构影响运营 | 中 | 先抽象兼容当前 `DataTable` 的 props，再逐页升级。 |
| 管理端移动端体验复杂 | 中 | 明确管理端以桌面为主，移动端只保障关键查看和轻操作。 |
| GitNexus 索引落后 | 低 | 本次为文档方案；后续编辑代码前应按 AGENTS 要求先做目标 symbol 的 impact。 |

## 16. 推荐落地优先级

1. **先做设计系统和 i18n 骨架**：这是后续页面商业化的地基。
2. **先改用户端快捷助手**：它是商业推广最容易成交的入口。
3. **再改管理端仪表盘和兑换码**：它们直接服务运营交付。
4. **最后整理发布、审计、自动化**：这些是高价值但低频的管理能力，重在风险表达。

最终目标不是把界面做得更“炫”，而是让 Tex2Doc 看起来像一个可以被购买、被交付、被运营、被信任的软件产品。
