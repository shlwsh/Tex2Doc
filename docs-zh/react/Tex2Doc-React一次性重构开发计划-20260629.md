# Tex2Doc React 一次性重构开发计划

> 更新日期：2026-06-29
> 基准方案：`Tex2Doc-React一次性重构Flutter用户端与管理端新版实现方案-20260629.md`
> 目标：新增 `apps/react-web`，以 React + TypeScript 一次性实现用户端与管理端 Web 闭环，开发期入口为 `/app-react` 与 `/admin-react`。

## 1. 开发边界

1. 不删除、不改造 Flutter Web 用户端和管理端，继续作为行为对照与回滚备份。
2. 不修改 Rust 后端接口语义，React 侧通过 TypeScript API SDK 对齐现有 `/v1/*` 与 `/admin/v1/*`。
3. 本阶段优先完成可运行、可联调、可验收的 React 实现；上线切换 `/app`、`/admin` 作为后续发布步骤。
4. 快捷助手本地转换必须遵守“先检查额度、转换成功后扣费、扣费成功后下载”的顺序。

## 2. 分阶段计划

### Phase 0：工程与契约

交付：

1. 创建 `apps/react-web` Vite + React + TypeScript 工程。
2. 建立路由、布局、主题、会话存储、通用表单/表格/状态组件。
3. 建立 API SDK，覆盖用户端、管理端、文件上传下载、兑换码、反馈、发布、自动化接口。
4. 建立 Vitest 基础测试，优先覆盖 API URL、错误模型、快捷助手扣费顺序。

验收：

1. `npm run typecheck` 通过。
2. `npm run build` 通过。
3. `/app-react`、`/admin-react`、`/react` 均可进入对应页面。

### Phase 1：管理端

交付：

1. 管理员登录、`/admin/v1/me` 校验、非管理员拒绝。
2. 仪表盘、账号摘要。
3. 兑换码生成、批次列表/详情/导出、库存筛选/分页/批量上货/导入重置/导出。
4. 反馈管理、状态修改、管理员回复。
5. 发布管理、必填校验、发布清单、回滚。
6. 审计中心。
7. 自动化研发面板，请求筛选、详情操作、Agent 暂停/恢复、自动刷新。

验收：

1. 管理端全导航可访问。
2. 核心写操作有确认、错误提示和成功刷新。
3. 导出类接口通过浏览器下载 Blob 文件。

### Phase 2：用户端会员中心

交付：

1. 登录、注册、退出、API Base URL 配置。
2. 账号与用量摘要。
3. 充值页，购买链接、兑换码输入、套餐/兑换记录。
4. 云端转换，上传 ZIP、创建任务、轮询、下载 DOCX。
5. 转换记录，DOCX/ZIP/LOG 下载。
6. 充值记录。
7. 反馈列表、创建、详情、继续回复。
8. 关于页、主题和语言偏好。

验收：

1. 会员登录后工作台刷新不丢失用户会话。
2. 云端转换轮询最多 120 次，失败状态清晰展示。
3. 反馈线程可创建、查看、回复。

### Phase 3：快捷助手

交付：

1. 兑换码激活与影子账号恢复。
2. 快捷版 ZIP 选择、主 TeX、Profile、Quality、WASM 懒加载转换。
3. 专业版云端转换。
4. 日志面板、DOCX 下载、用量刷新。

验收：

1. 有效兑换码可激活并写入 `tex2doc.quick.redeemCode`。
2. 刷新后自动恢复快捷助手会话。
3. 额度不足不转换，转换失败不扣费，扣费失败不下载。

### Phase 4：部署切换

交付：

1. 增加 React 静态构建脚本，输出 home/app/admin 三类产物。
2. React 与 Flutter 走同一验收清单。
3. React 通过后，将正式路径 `/app` 与 `/admin` 切到 React。
4. Flutter 构建产物保留为回滚备份。

验收：

1. 用户端与管理端 E2E 通过。
2. 首屏和管理端交互性能满足方案指标。
3. 回滚路径可恢复 Flutter Web 产物。

## 3. 当前实现顺序

1. Phase 0 工程骨架、API SDK、通用布局。
2. Phase 1 管理端完整页面与接口联调入口。
3. Phase 2 会员中心页面。
4. Phase 3 快捷助手页面与本地/云端转换流程。
5. 基础测试、构建验证、变更影响检查。

## 4. 风险控制

1. API 字段可能存在历史返回格式差异，React SDK 使用宽松 DTO 并在页面层做空值兜底。
2. WASM 产物可能尚未构建，快捷版本地转换先提供懒加载适配层，缺失时明确提示运行 `npm run build:wasm:react`。
3. 管理端高风险自动化请求不显示自动批准按钮，只允许升级人工。
4. GitNexus 当前 MCP 工具未暴露 `impact`，本次优先新增文件；如后续修改既有符号，应先补跑可用的影响分析或重新暴露工具。
