# Tex2Doc 四部分发布形态工程落地实现报告

生成日期：2026-06-24

## 1. 实现结论

已完成第一阶段工程落地：项目结构已具备四个发布单元边界，Flutter Web 已具备产品首页、用户端、管理端三类入口，服务端已支持 Web 静态资源托管和 `/admin/v1/me` 管理身份校验，用户端导航已与管理能力分离。

本轮未进行大规模物理迁移 Rust crate 或 Slint crate，采用方案建议的低风险方式：先建立 `apps/` 发布单元映射、Flutter 多入口 target 和服务端托管能力，后续再逐步拆分 Flutter 工程与移动 Rust/Slint 发布单元。

## 2. 已完成内容

### 2.1 四发布单元目录

新增：

- `apps/rust-service`
- `apps/flutter-admin`
- `apps/flutter-user`
- `apps/slint-user`

其中 `apps/rust-service/static/home`、`apps/rust-service/static/user`、`apps/rust-service/static/admin` 作为服务端静态资源部署目标目录。

### 2.2 Flutter Web 三入口

新增：

- `flutter_app/lib/main_user.dart`
- `flutter_app/lib/main_admin.dart`
- `flutter_app/lib/product/product_home_app.dart`

调整：

- `flutter_app/lib/main.dart` 根据 URL path 分流：
  - `/`：产品介绍首页。
  - `/app`：Flutter 用户端。
  - `/admin`：Flutter 管理端。
- `DocEngineApp` 增加 `DocEngineAppMode.user/admin`。
- 用户端导航仅保留用户相关模块。
- 管理端独立启动，非管理员登录时提示 `Admin role required.`。
- 管理端反馈模块已独立为 `AdminFeedbackPanel`，调用 `/admin/v1/feedback/threads`，支持列表、刷新、状态更新和管理员回复。
- 管理端兑换码批次管理和记录页已使用管理端登录 token，不再要求管理员在页面内二次输入 `demo-admin` token。

### 2.3 服务端 Web 托管与管理身份

调整：

- `doc-server` 启用 `tower-http/fs`。
- 新增静态托管根目录配置：`TEX2DOC_STATIC_DIR`，默认 `apps/rust-service/static`。
- 服务端托管：
  - `/` -> `static/home/index.html`
  - `/app/*` -> `static/user`
  - `/admin/*` -> `static/admin`
  - `/assets/*` -> `static/assets`
- 新增 `/admin/v1/me`。
- `require_admin_session` 支持数据库 Bearer token + role 校验，同时保留 `demo-admin` preview token。
- `AppUser` 与 API JSON 返回中加入 `role`。
- `admin@example.com` + `demo-admin` 登录会生成管理员访问 token。

### 2.4 构建脚本

新增：

- `scripts/release/build-flutter-home.ps1`
- `scripts/release/build-rust-service.ps1`
- `scripts/release/build-flutter-admin.ps1`
- `scripts/release/build-flutter-user.ps1`
- `scripts/release/build-slint-user.ps1`
- `scripts/release/package-web-static.ps1`

其中 Flutter 首页、用户端、管理端构建脚本会在构建完成后复制 `flutter_app/build/web` 到：

- `apps/rust-service/static/home`
- `apps/rust-service/static/user`
- `apps/rust-service/static/admin`

## 3. 验证结果

已通过：

```text
cargo check -p doc-server
cargo test -p doc-server --test api -- --nocapture --test-threads=1
cargo test -p doc-desktop-slint
cargo test -p doc-commercial-api-client
flutter analyze
flutter test
flutter build web --target lib/main_user.dart
flutter build web --target lib/main_admin.dart
```

`doc-server` API 测试已从 14 项增加到 15 项，新增覆盖：

- 普通用户访问 `/admin/v1/me` 返回 401。
- `demo-admin` preview token 可访问 `/admin/v1/me`。
- `admin@example.com` + `demo-admin` 登录 token 可访问 `/admin/v1/me`。

## 4. GitNexus 影响分析

开发前已对关键符号执行 impact：

- `main`
- `DocEngineApp`
- `router_with_state`
- `CommercialApiClient`
- `require_admin_session`
- `app_user_json`
- `UserProfile`
- `AppUser`
- `auth_response`

其中 `CommercialApiClient` 为 CRITICAL，`require_admin_session` 为 HIGH，其余多为 LOW/MEDIUM。本轮对这两个高影响点均采用向后兼容加法：

- `CommercialApiClient` 新增 `adminMe` 与 `AdminProfile`，不改变原有方法签名。
- `require_admin_session` 保留旧 preview token，同时新增数据库 role 校验。

开发后已执行 `detect_changes(scope=all)`，结果为 high，符合管理鉴权、Web 入口和共享 API client 改造的预期影响范围。

## 5. 当前边界

已完成第一阶段和部分第二阶段能力，但仍未完成以下中长期事项：

1. 未将 `crates/server` 物理移动到 `apps/rust-service`。
2. 未将 `crates/desktop-slint` 物理移动到 `apps/slint-user`。
3. 未将 `flutter_app` 物理拆分为两个独立 Flutter 工程。
4. 管理端仍复用现有工作台部分页面，尚未补齐完整后台模块页面。
5. 产品首页按钮在 Flutter 内部切换应用模式；生产部署中直接访问 `/app`、`/admin` 已由入口分流支持。

## 6. 建议下一步

建议继续推进：

1. 将现有 `workspace_app.dart` 拆成 `user/`、`admin/`、`shared/`。
2. 将管理端兑换码、反馈、发布、审计等页面独立成 `admin/pages/*`。
3. 新增管理端 dashboard 聚合 API。
4. 将 Flutter 构建产物复制进 `apps/rust-service/static/*` 的发布脚本。
5. 后续再进行 Rust/Slint 发布单元物理迁移。
