# Tex2Doc 四部分工程组织对齐规划实现方案
> **版本 / Version**: v2.0
> **最后更新日期 / Last Updated**: 2026-06-26



生成日期：2026-06-24  
关联文档：`docs-zh/home/Tex2Doc最终发布形态与服务端Web双入口设计实现方案-20260624.md`  
目标：将当前项目工程组织与最终发布形态对齐，形成四个清晰发布单元：Rust 服务/核心转换引擎、Flutter 管理端、Flutter 用户端、Slint 用户端。

## 1. 总体结论

Tex2Doc 最终不应只按技术目录划分，而应按“可发布产品单元”划分。建议将工程组织调整为：

1. **Rust 服务 - 核心转换引擎**
   - 发布物：`tex2doc-server` 服务端二进制、Docker 镜像、数据库迁移脚本。
   - 职责：API、认证、商业化、PostgreSQL、文件存储、worker、核心转换调用。

2. **Flutter 管理端**
   - 发布物：管理端 Web 静态资源。
   - 职责：运营后台、用户/套餐/订单/兑换码/反馈/发布/审计管理。

3. **Flutter 用户端**
   - 发布物：用户端 Web 静态资源。
   - 职责：用户登录、转换、记录、充值、兑换、反馈、桌面端下载。

4. **Slint 用户端**
   - 发布物：Windows/macOS/Linux 桌面安装包。
   - 职责：本地转换、云端转换、账号、账单、历史、反馈、自动更新。

建议新增顶层 `apps/` 作为发布应用目录，保留 `crates/` 作为 Rust 共享库目录。这样既能和最终发布形态一致，又能降低一次性迁移风险。

## 2. 当前工程与目标工程对照

| 当前位置 | 当前职责 | 目标归属 | 调整建议 |
|---|---|---|---|
| `crates/server` | Rust API 服务端 | Rust 服务 - 核心转换引擎 | 迁移或映射为 `apps/rust-service` |
| `crates/core`、`compiler-engine`、`docx-writer` 等 | 转换核心库 | Rust 服务 - 核心转换引擎共享库 | 保留在 `crates/` |
| `crates/wasm`、`crates/native` | Flutter/本地桥接能力 | 共享桥接层 | 保留在 `crates/`，由用户端按需依赖 |
| `crates/commercial-api-client` | 商业 API Rust client | Slint 用户端共享依赖 | 保留在 `crates/` |
| `crates/desktop-slint` | Slint 桌面端 | Slint 用户端 | 迁移或映射为 `apps/slint-user` |
| `flutter_app` | Flutter Web 混合入口 | Flutter 管理端 + Flutter 用户端 | 拆为 `apps/flutter-admin` 与 `apps/flutter-user` |
| `docs-zh/money` | 商业化数据库/方案 | Rust 服务支撑文档 | 保留，后续可归档到 `docs-zh/server` |
| `scripts` | 构建、验证、发布脚本混杂 | 各发布单元脚本 | 分阶段拆到 `apps/*/scripts` 和 `scripts/release` |

## 4. Rust 服务 - 核心转换引擎组织方案

### 4.1 目标职责

Rust 服务单元应包含：

- Axum HTTP API。
- PostgreSQL 持久化。
- 转换任务 DB queue。
- worker 调度。
- 文件存储。
- 商业化能力：用户、套餐、订单、兑换码、用量、反馈、发布版本。
- 调用核心转换引擎完成 TeX/LaTeX 到 DOCX 的转换。
- 静态托管 Flutter 首页、用户端和管理端构建产物。

### 4.2 推荐 crate 边界

```text
apps/rust-service
  -> depends on crates/core
  -> depends on crates/compiler-engine
  -> depends on crates/docx-writer
  -> depends on crates/tex-facade
  -> depends on crates/quality

crates/*
  -> 保持纯库职责
  -> 不直接读取 Web session
  -> 不直接访问 PostgreSQL
```

这样服务端是应用入口，核心转换引擎是可复用库。未来 Slint 本地转换、WASM 转换也能继续复用核心库。

### 4.3 迁移建议

短期建议先不移动所有 Rust crate，只做两步：

1. 将 `crates/server` 标记为 Rust 服务发布单元，并补充 `apps/rust-service/README.md` 或 `apps/rust-service` wrapper。
2. 后续稳定后再把 `crates/server` 物理移动到 `apps/rust-service`。

这样可以避免 Cargo workspace、GitNexus 索引、测试路径和 CI 一次性大规模变更。

## 5. Flutter 管理端组织方案

### 5.1 目标职责

管理端只服务管理员、运营和客服，模块包括：

- 概览仪表盘。
- 用户管理。
- 套餐管理。
- 充值/订单管理。
- 兑换码批次管理。
- 转换任务管理。
- 反馈管理。
- 发布版本管理。
- 审计日志。
- 系统设置。

### 5.2 目录建议

### 5.3 API 约束

管理端只能调用：

- `/admin/v1/*`
- 必要时调用 `/v1/auth/login` 获取 token。
- 登录后必须调用 `/admin/v1/me` 校验管理员身份。

不建议管理端直接复用用户端业务页面，避免权限边界模糊。

## 6. Flutter 用户端组织方案

用户端只服务普通用户，模块包括：

- 账号资料。
- 用量与权益。
- 套餐展示、充值、兑换码使用。
- 转换任务创建。
- 转换记录、下载、报告。
- 反馈提交和回复查看。
- Slint 桌面端下载入口。

### 6.3 API 约束

用户端只能调用：

- `/v1/*`
- `/api/v1/*` 兼容路径可保留，但新代码优先使用 `/v1/*`。

用户端不应包含：

- `/admin/v1/*` API client。
- 管理端菜单。
- 运营后台页面。


共享包只能放通用能力，不放具体业务页面。尤其要避免把管理端 API client 放进用户端可访问路径。

短期可以从当前 `flutter_app/lib/ui`、`commercial_api.dart` 中逐步拆出：

- 主题。
- 国际化。
- 通用组件。
- HTTP 基础封装。
- 用户模型和错误模型。

## 9. 产品首页归属

产品介绍首页不属于四个登录后应用之一，但需要作为服务端 Web 发布的一部分托管。建议两种方式：

### 9.1 短期方案

将产品首页放入 `apps/flutter-user`：

- `/` 产品首页。
- `/app` 用户端工作台。

优点：改造量小，用户入口与产品营销入口联系紧密。

### 9.2 中长期方案

独立为：

```text
apps/site-home/
```

优点：官网可独立发版、可使用更适合内容站的技术栈。  
当前不建议立即新增第五个发布单元，以免偏离“四部分”主目标。

## 12. 分阶段迁移计划

### 阶段一：建立发布单元边界

- 新增 `apps/` 目录。
- 为四个发布单元建立 README 和构建说明。
- 将当前路径映射写清楚：
  - `crates/server` -> `apps/rust-service`
  - `flutter_app` 用户模块 -> `apps/flutter-user`
  - `flutter_app` 管理模块 -> `apps/flutter-admin`
  - `crates/desktop-slint` -> `apps/slint-user`
- 不立即大规模移动代码，先减少路径变更风险。

### 阶段二：Flutter 逻辑拆分

- 在当前 `flutter_app` 中新增 `main_user.dart`、`main_admin.dart`。
- 将 `workspace_app.dart` 拆为 `user/`、`admin/`、`shared/`。
- 用户端移除管理模块。
- 管理端接入现有 `/admin/v1/*` 模块。
- 抽出 `packages/flutter-shared`。

### 阶段三：Flutter 物理拆分

- 创建 `apps/flutter-user`。
- 创建 `apps/flutter-admin`。
- 将共享组件迁入 `packages/flutter-shared`。
- 两个 Flutter app 分别维护 `pubspec.yaml`、`web/index.html`、测试和构建配置。
- E2E 测试分别覆盖 `/app` 与 `/admin`。

### 阶段四：Rust 发布单元物理调整

- 将 `crates/server` 迁移到 `apps/rust-service`，或保留路径但 rename package 为 `tex2doc-server`。
- 更新 `Cargo.toml` workspace members。
- 更新 CI、脚本、Dockerfile、GitNexus 索引。
- 保持 `crates/*` 作为纯库。

### 阶段五：Slint 发布单元物理调整

- 将 `crates/desktop-slint` 迁移到 `apps/slint-user`，或保留路径但 rename package 为 `tex2doc-slint-user`。
- 补齐 `packaging/windows`、`packaging/macos`、`packaging/linux`。
- 与 release manifest、自动更新链路对齐。

## 13. CI/CD 调整建议

CI 建议按发布单元拆 job：

| Job | 校验内容 |
|---|---|
| `rust-service-check` | `cargo check -p tex2doc-server`、server tests |
| `rust-core-check` | 核心转换 crates 单元测试和回归测试 |
| `flutter-user-check` | Flutter analyze、用户端 widget/e2e |
| `flutter-admin-check` | Flutter analyze、管理端 widget/e2e |
| `slint-user-check` | `cargo test -p tex2doc-slint-user` |
| `package-release` | 组合构建 dist 产物 |

发布流水线建议输出：

- 服务端 Docker 镜像。
- 用户端 Web 静态包。
- 管理端 Web 静态包。
- 桌面端安装包。
- 数据库迁移包。
- release manifest。

## 14. 代码迁移风险与控制

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 一次性移动路径过多 | 高 | 先建立 `apps/` 映射，再分阶段物理迁移 |
| Cargo workspace 成员变化导致测试失效 | 中 | 每次只迁移一个 Rust 发布单元，迁移后跑 cargo check/test |
| Flutter 管理端/用户端共享代码复制 | 中 | 抽 `packages/flutter-shared`，禁止页面级复用 |
| 用户端误引入管理 API | 高 | 拆分 `user_api_client` 与 `admin_api_client`，lint/测试检查 `/admin/v1` |
| Slint 路径迁移影响资源引用 | 中 | 先保留原资源结构，迁移后重点验证 `slint` UI assets |
| 发布脚本和 CI 路径过期 | 高 | 迁移阶段同步更新 `scripts/release` 和 `.github/workflows` |

## 15. 建议优先级

推荐先做“低风险边界建立”，再做物理迁移：

1. 新增 `apps/` 发布单元说明，不移动代码。
2. Flutter 先拆入口和模块，再拆工程。
3. Rust 服务先完成静态托管和 API 分域，再考虑移动 `crates/server`。
4. Slint 在发布打包稳定后再从 `crates/desktop-slint` 移动到 `apps/slint-user`。
5. 每完成一个阶段都更新 CI、构建脚本和文档。

## 16. 新版落地判断

完成后，项目根目录应能让新成员一眼看出四个产品边界：

```text
apps/rust-service      服务端与核心转换发布入口
apps/flutter-admin     Web 管理端
apps/flutter-user      Web 用户端
apps/slint-user        桌面用户端
crates                 Rust 共享库
packages/flutter-shared Flutter 共享库
```

这会让后续发布部署、权限隔离、CI/CD、测试矩阵和商业化演进都更清楚。当前最适合先输出 `apps/` 边界与 Flutter 双入口拆分，等功能稳定后再进行 Rust 和 Slint 的物理目录迁移。
