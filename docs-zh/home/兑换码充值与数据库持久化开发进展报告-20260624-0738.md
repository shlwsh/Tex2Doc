## 1. 开发目标

本轮工作的主目标是将 Flutter / Slint 客户端的商业化充值链路从预览态推进到可持久化、可审计的本地 `docdb` 数据库闭环。

核心目标包括：

1. 兑换码模式替代 mock 充值。
   - 服务端批量生成兑换码。
   - 兑换码绑定指定转换套餐。
   - 每个兑换码必须有数据库记录。
   - 支持 Excel 批量导出下载。
   - 用户兑换后写入充值记录和权益余额。
   - 兑换码一经核销即作废，禁止重复使用。

2. Web / Flutter 应用补齐运营管理入口。
   - 左侧导航增加“兑换码管理”。
   - 允许管理员生成兑换码批次。
   - 允许导出当前批次 Excel。
   - 后续新增“兑换记录”模块，可按批次查看历史生成记录和导出。

3. Slint 桌面端同步充值和记录查询能力。
   - 支持兑换码核销。
   - 支持充值记录、转换记录以表格方式展示。
   - 修复 demo 登录失败和端口冲突问题。

4. 服务端必须全部接入 PostgreSQL `docdb`。
   - 不允许业务数据只存在内存。
   - 用户、token、用量、上传、转换任务、充值、兑换码、反馈等模块均需落盘。
   - 服务端启动应依赖数据库连接，连接失败应失败退出，而不是回退内存。

## 6. 当前主要风险

### 6.1 服务端处于数据库迁移中间态

`state.rs` 已开始移除内存 HashMap 并接入 `DbStore`，但还未完整替换所有调用点。

风险：

- 可能存在编译错误。
- 路由层仍有部分 demo auth 逻辑尚未完全接 DB。
- Worker 读取上传文件和任务状态的逻辑需要与 DB object_key / session 文件路径对齐。

### 6.2 数据库 schema 与运行时模型存在差异

已有 schema 是设计基线，但运行时代码还需要补齐：

- `commercial_entitlements` 当前作为新增辅助表，用于保存兑换码余额和有效期。
- `app_access_tokens` 当前作为新增辅助表，用于 token 到 user 的映射。
- `conversion_jobs.result_report_key` 当前可能复用 JSON 字符串保存 report，不够干净，后续建议增加独立 report 表或 report object_key。

### 6.3 反馈模块仍是内存实现

`feedback_service.rs` 当前明确写着 in-memory。

用户要求下，它必须改为 PostgreSQL：

- `feedback_threads`
- `feedback_messages`

### 6.4 上传/转换文件需要明确“落盘”边界

推荐边界：

- 文件本体落本地 `sessions/` 目录。
- PostgreSQL 保存：
  - object_key
  - sha256
  - bytes
  - status
  - job_id
  - result_docx_key
  - result_log_key

不能只把 ZIP/DOCX bytes 放在进程内存中。

### 6.5 Worker 队列是否必须持久化需要确认

用户强调“不允许在内存中空存”。如果严格解释为任务队列也必须持久化，则 `mpsc` 也需要替换。

建议下一阶段至少做到：

- 任务创建写入 `conversion_jobs.status='queued'`。
- Worker 定时/循环从 DB 拉取 queued job。
- Worker 更新状态到 DB。
- 服务重启后可恢复未完成任务。
