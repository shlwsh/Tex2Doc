# 反馈模块与增强型切换记录技术实现方案

生成时间：2026-06-24

## 1. 背景与目标

当前系统现状：
- 服务端已具备 in-memory 的 `ServerState`，管理 `uploads`、`jobs`、`recharges` 等记录。
- Flutter 应用 (`flutter_app/`) 和 Slint 桌面应用 (`crates/desktop-slint/`) 均通过 `CommercialApiClient` 与服务端交互。
- PostgreSQL schema (`001_docdb_business_schema.sql`) 已定义 `conversion_jobs`、`uploads` 表。
- 当前 ZIP 上传后存 in-memory，未持久化到分级目录。

本次实现目标：

1. **反馈模块** — 用户提交问题/需求，必须关联具体转换记录，服务端存储；管理端可回复，用户可继续反馈，形成 Chat 模式交互；管理端支持导出申请内容为 Excel 并下载。

2. **增强型切换记录** — 服务端将 ZIP 上传文件、转换结果 DOCX、转换日志存储到 `sessions/{年}/{月}/{日}/{job_id}/` 分级目录，同步数据库 `conversion_jobs` 表；前端展示并允许下载这三个文件。

## 2. 数据库设计

### 2.1 增强 `conversion_jobs` 表

在现有 `conversion_jobs` 表基础上，新增字段（通过 migration）：

```sql
ALTER TABLE conversion_jobs
    ADD COLUMN IF NOT EXISTS source_zip_key TEXT,
    ADD COLUMN IF NOT EXISTS result_log_key TEXT,
    ADD COLUMN IF NOT EXISTS storage_path TEXT,
    ADD COLUMN IF NOT EXISTS zip_bytes BIGINT,
    ADD COLUMN IF NOT EXISTS docx_bytes BIGINT,
    ADD COLUMN IF NOT EXISTS log_bytes BIGINT;
```

### 2.2 新增反馈主题表

```sql
CREATE TABLE IF NOT EXISTS feedback_threads (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES app_users(id) ON DELETE CASCADE,
    conversion_job_id UUID REFERENCES conversion_jobs(id) ON DELETE SET NULL,
    title TEXT NOT NULL,
    feedback_type TEXT NOT NULL CHECK (feedback_type IN ('issue', 'requirement')),
    status TEXT NOT NULL DEFAULT 'open'
        CHECK (status IN ('open', 'in_progress', 'resolved', 'closed')),
    priority TEXT NOT NULL DEFAULT 'normal'
        CHECK (priority IN ('low', 'normal', 'high', 'urgent')),
    admin_assignee UUID REFERENCES app_users(id) ON DELETE SET NULL,
    message_count INT NOT NULL DEFAULT 0,
    latest_message_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_feedback_threads_user
    ON feedback_threads(user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_feedback_threads_job
    ON feedback_threads(conversion_job_id) WHERE conversion_job_id IS NOT NULL;
CREATE INDEX IF NOT EXISTS idx_feedback_threads_status
    ON feedback_threads(status, created_at DESC);
```

### 2.3 新增反馈消息表

```sql
CREATE TABLE IF NOT EXISTS feedback_messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    thread_id UUID NOT NULL REFERENCES feedback_threads(id) ON DELETE CASCADE,
    parent_message_id UUID REFERENCES feedback_messages(id) ON DELETE SET NULL,
    sender_user_id UUID REFERENCES app_users(id) ON DELETE SET NULL,
    sender_type TEXT NOT NULL CHECK (sender_type IN ('user', 'admin', 'system')),
    content TEXT NOT NULL,
    attachments JSONB NOT NULL DEFAULT '[]'::jsonb,
    is_internal BOOLEAN NOT NULL DEFAULT false,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_feedback_messages_thread
    ON feedback_messages(thread_id, created_at ASC);
```

## 3. 服务端文件存储设计

### 3.1 目录结构

```
{sessions_root:-./sessions}/
└── sessions/
    └── {YYYY}/
        └── {MM}/
            └── {DD}/
                └── {job_id}/
                    ├── source.zip          # 用户上传的 ZIP 文件
                    ├── result.docx        # 转换生成的 DOCX 文件
                    └── conversion.log      # 转换过程日志
```

### 3.2 核心模块

**`crates/server/src/file_storage.rs`**：
- `FileStorage` 结构体，使用 `chrono` 获取当前日期
- `session_dir()` — 构建日期分级目录路径
- `store()` — 写入文件到 session 目录
- `load()` — 读取 session 文件
- `exists()` — 检查文件是否存在
- `file_key()` — 返回相对路径字符串

**`crates/server/src/feedback_service.rs`**：
- `FeedbackStore` — 线程安全反馈存储（基于 `Arc<RwLock>`）
- `FeedbackThread` / `FeedbackMessage` / `FeedbackThreadSummary` 数据结构
- `FeedbackType` / `FeedbackStatus` / `FeedbackPriority` / `SenderType` 枚举
- `FeedbackError` 错误类型
- CRUD 操作：`create_thread`、`add_message`、`admin_reply`、`admin_update`、`list_user_threads`、`admin_list`、`get_thread_for_user`

**`crates/server/src/excel_export.rs`**：
- `build_feedback_export_xlsx()` — 生成 OOXML 格式 Excel
- `build_admin_feedback_export()` — 管理端导出（含线程列表和消息详情）
- 复用 `zip` crate 手写 XML 策略

## 4. 服务端 API 设计

### 4.1 用户端反馈接口

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/v1/feedback/threads` | 创建反馈主题 |
| `GET` | `/v1/feedback/threads` | 获取反馈线程列表 |
| `GET` | `/v1/feedback/threads/:id` | 获取反馈线程详情 |
| `POST` | `/v1/feedback/threads/:id/messages` | 追加反馈消息 |

### 4.2 管理端反馈接口

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/admin/v1/feedback/threads` | 获取所有反馈（支持分页过滤） |
| `PATCH` | `/admin/v1/feedback/threads/:id` | 更新反馈状态/优先级 |
| `POST` | `/admin/v1/feedback/threads/:id/messages` | 管理端回复 |
| `GET` | `/admin/v1/feedback/threads/export.xlsx` | 导出 Excel |

### 4.3 转换记录文件下载接口

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/v1/conversions/:id/download/zip` | 下载源 ZIP |
| `GET` | `/v1/conversions/:id/download/log` | 下载转换日志 |
| `GET` | `/v1/conversions/:id` | 获取转换详情（含 storage_info） |

## 9. 关键实现细节

### 9.1 反馈 Chat 模型

- 每条反馈线程初始包含一条用户消息（`message_count=1`）
- 管理端首次回复时自动将状态从 `open` 变更为 `in_progress`
- `message_count` 和 `latest_message_at` 在每次消息追加时更新
- 用户只能访问自己的反馈线程；管理端可访问所有

### 9.2 文件存储安全

- `sanitize_id()` 过滤 `job_id`，只允许 `a-zA-Z0-9_-`
- `fixed_filename()` 限制文件名只能是 `source.zip`、`result.docx`、`conversion.log`
- 使用 `PathBuf::push` 构建路径，禁止字符串拼接

### 9.3 Excel 导出格式

导出包含两个 Sheet：
1. **反馈线程列表** — ID、用户、转换记录、类型、标题、状态、优先级、消息数、最新回复时间、创建时间
2. **消息详情** — 线程 ID、发送者类型、内容、时间
