# Tex2Doc 产品研发待推进工作核查与技术实施方案

生成日期：2026-06-24  
核查方式：产品商业化评估技能 + GitNexus 代码索引 + 现有中文规划文档 + 本地代码结构抽样核查  
适用范围：Tex2Doc 商业化 Preview 到邀请制 Beta 的研发推进计划  

## 1. 总体结论

按产品商业化成熟度 rubric 判断，Tex2Doc 当前仍处于**可试点阶段**：核心 LaTeX/TeX 到 DOCX 转换价值已经成立，服务端、Flutter Web、Slint 桌面端、兑换码、反馈、升级检查、文件落盘、PostgreSQL 主链路都已有工程基础；但距离可商业化阶段还差“可恢复、可审计、可交付、可升级、可支持”的生产闭环。

本次核查后的核心结论是：

1. **服务端商业主链路已经从内存态推进到 PostgreSQL 态，但仍需要生产语义收口。**  
   `apps/rust-service/src/state.rs` 当前已以 `DbStore + FileStorage + FeedbackStore + mpsc notify` 为主结构，`apps/rust-service/src/db_store.rs` 已出现 `claim_next_job`、`recover_stale_jobs`、`usage_ledger`、`report_json` 等关键能力。下一步不是从零开发，而是完成语义严谨性、测试和交付门禁。

2. **用户、管理员、产品首页三入口已开始拆分，但管理端仍偏骨架。**  
   `flutter_app/lib/main.dart` 已按 `/admin`、`/app`、`/` 分流，`product/`、`user/`、`admin/` 目录已经存在。但多个后台模块仍需要真实数据、权限、分页过滤、审计和验收覆盖。

3. **商业化阻塞点从“能不能做”转为“能不能放心给真实客户用”。**  
   当前最大风险不是核心转换引擎，而是 demo auth、mock pay、发布升级管理、支付/账本、文件与任务恢复、质量证明、安装包与获客入口这些产品工程环节。

## 4. P0 技术实施方案

### 4.1 生产级认证与权限

#### 当前问题

`apps/rust-service/src/routes.rs` 仍存在明显 demo 行为：

- `auth_register` / `auth_login` 缺省 email 为 `demo@example.com`。
- admin 登录仍识别 `admin@example.com` + `demo-admin`。
- token 前缀仍使用 `demo-access`、`demo-refresh`。
- `require_admin_session` 对 `admin-` 和 `demo-admin` 有特殊通道。

这些逻辑适合本地 Preview，但不适合邀请制 Beta。

#### 实施方案

1. 在 `app_users` 中明确账号状态字段：
   - `role`: `user` / `admin` / `operator` / `support`
   - `status`: `active` / `disabled` / `pending`
   - `password_hash`
   - `last_login_at`

2. 将 auth 路由改为真实语义：
   - `POST /v1/auth/register`：重复邮箱返回稳定错误。
   - `POST /v1/auth/login`：校验密码 hash，失败返回 401。
   - `POST /v1/auth/refresh`：refresh token hash + rotation + revoke old token。
   - `POST /v1/auth/logout`：撤销当前 access/refresh token。

3. 管理端统一使用 DB role：
   - 移除 `demo-admin` 和 `admin-` 后门。
   - `/admin/v1/me` 返回角色、权限集合、可访问模块。
   - 管理端 session 使用独立 localStorage key。

4. 补充测试：
   - 普通用户访问 `/admin/v1/me` 返回 403。
   - disabled 用户无法登录。
   - refresh token 被轮换后旧 token 不可再用。
   - 管理员降权后旧 session 失效或下次校验失败。

#### 验收标准

- 服务端代码中不再出现 `demo-admin` 作为鉴权通道。
- 所有管理 API 都通过 DB role 判断权限。
- `cargo test -p doc-rust-service` 或实际包名对应测试通过。

### 4.2 PostgreSQL 主链路收口

#### 当前问题

`ServerState` 已经基本切到 `DbStore`，但仍需要做生产级审计：

- `UploadRecord` / `ConversionJobRecord` 仍保留 `bytes`、`docx` 等内存字段作为模型字段。
- 文件本体应只在请求或 worker 执行期间作为局部变量存在。
- 辅助表和 migration 需要统一，避免 `DbStore::init_schema` 与 SQL 文件双轨。
- 文档中旧路径多处写 `crates/server`，实际服务端为 `apps/rust-service`。

#### 实施方案

1. 建立正式 migration 基线：
   - 合并 `docs-zh/money/001_docdb_business_schema.sql`
   - 合并 `docs-zh/money/003_feedback_and_session_storage.sql`
   - 纳入 `app_access_tokens`
   - 纳入 `commercial_entitlements`
   - 纳入 `usage_ledger`
   - 纳入 `conversion_jobs.report_json`、`worker_id`、`locked_at`、`attempts`、`next_run_at`

2. 明确 `DbStore::init_schema` 职责：
   - 只执行 migration 或校验 schema 版本。
   - 不再悄悄创建生产表结构。

3. 扫描并分类内存状态：
   - 允许：连接池、配置、局部变量、worker notify。
   - 不允许：业务事实源、任务队列、权益余额、反馈记录、兑换码、上传记录。

4. 修正文档和代码路径：
   - 新方案统一使用 `apps/rust-service`。
   - 若旧 crate 名仍有残留，建立迁移备注，避免任务误派。

#### 验收标准

- PostgreSQL 不可用时服务启动失败。
- 服务重启后用户、token、上传、任务、反馈、兑换码、用量、release manifest 均可恢复。
- 业务表结构只通过 migration 管理。

### 4.3 转换任务 DB Queue 与重启恢复

#### 当前进展

`apps/rust-service/src/worker_service.rs` 已经体现正确方向：

- `mpsc` 只接收 `WorkerCommand` 作为唤醒。
- worker 定时调用 `recover_stale_jobs()`。
- worker 循环通过 `claim_next_job(worker_id)` 领取任务。

这说明 DB queue 方向已启动，下一步是做完整恢复和幂等验证。

#### 实施方案

1. 完成 `conversion_jobs` 队列字段：
   - `status`
   - `worker_id`
   - `locked_at`
   - `attempts`
   - `next_run_at`
   - `queued_at`
   - `started_at`
   - `completed_at`
   - `failed_at`

2. 领取任务使用 PostgreSQL row lock：
   - `FOR UPDATE SKIP LOCKED`
   - 多 worker 不重复领取。
   - processing 超时后可重新 queued。

3. worker 状态机固定为：
   - `queued`
   - `normalizing`
   - `detecting`
   - `analyzing`
   - `compiling`
   - `rendering`
   - `verifying`
   - `completed`
   - `failed`

4. 增加重启恢复 E2E：
   - 创建 job 后停止服务。
   - 重启后 worker 能继续处理 queued job。
   - processing 超时 job 能按 backoff 恢复或失败。
   - completed job 重启后仍能下载 DOCX/log/report。

#### 验收标准

- 任务不依赖内存队列保存事实状态。
- 同一个 job 不会因重试重复扣费。
- worker 崩溃后任务不会永久卡在 processing。

### 4.4 用量账本、失败返还与订单审计

#### 当前问题

`usage_ledger` 已在 `db_store.rs` 中出现，说明研发已经开始补账本。但还需要把“预占、确认、返还、人工调整”形成稳定产品语义。

#### 实施方案

1. 定义 ledger 事件：
   - `grant`：兑换码、人工发放、活动赠送。
   - `reserve`：创建转换任务时预占额度。
   - `commit`：转换成功后确认消耗。
   - `refund`：转换失败、取消、超时后返还。
   - `adjust`：管理员修正。

2. 增加幂等约束：
   - `conversion_job_id + event_type` 唯一或业务唯一索引。
   - 重试时重复 `commit` / `refund` 不重复影响余额。

3. 前端展示用量结构：
   - 剩余转换次数。
   - 预占中次数。
   - 本周期已用。
   - 兑换码获得次数。
   - 最近账本明细。

4. 管理端补账本查询：
   - 按用户查。
   - 按 job 查。
   - 按来源查。
   - 支持导出。

#### 验收标准

- 成功转换只扣一次。
- 失败转换自动返还。
- 客服能通过 job id 查清扣费原因。
- 管理端可以补偿发放并留下审计。

#### 当前问题

产品价值已经能说明，但外部用户要成功试用还需要安装、示例、手册、失败恢复路径。

   - 加版本号。

3. 诊断包：
   - app version。
   - server version。
   - job id。
   - error code。

   - report json。
   - 脱敏文件清单。

#### 验收标准

- 失败时能导出诊断包并提交反馈。
- 支持人员可通过 job id 复现或定位。

### 60 天：邀请制 Beta / 付费 Beta 准备

目标：基于数据决定是否扩大获客。

## 9. 风险与控制

| 风险 | 等级 | 控制措施 |
|---|---|---|
| demo auth 残留进入 Beta | 高 | 去除特殊 token；所有权限走 DB role；补权限测试 |
| DB schema 双轨 | 高 | migration 统一；`init_schema` 只跑 migration |
| worker 重复消费 | 高 | `FOR UPDATE SKIP LOCKED`；job 幂等；测试并发 worker |
| 失败重复扣费 | 高 | ledger 唯一约束；reserve/commit/refund 幂等 |
| 文件跨日或迁移后不可下载 | 中 | 全部按 DB object key 读取，不按日期反推 |
| 管理端全量加载 | 中 | SQL 分页过滤；加索引；导出走流式或后台任务 |
| 发布坏版本无法回滚 | 高 | release strategy + rollback API + 审计 |
| 用户无法独立试用 | 高 | installer、demo、手册、FAQ、诊断包一起交付 |

## 10. 下一步建议

建议立即创建 4 条并行研发线：

1. **服务端生产化线**：auth、migration、DB queue、ledger、恢复测试。
2. **管理端线**：dashboard、兑换码、反馈、用户、账本、release。
3. **客户端交付线**：installer、升级提示、诊断包、demo 包。
4. **商业入口线**：landing page、waitlist、下载页、价格/套餐和试用材料。

其中第一条线是所有后续商业化工作的地基，应作为 P0 每日收敛；其余三条可以在不破坏主链路的前提下并行推进。当前最适合的产品目标不是公开 GA，而是**30 天内完成受控 PoC，60 天内进入邀请制 Beta 准备，90 天内基于真实数据决定是否公开 Beta**。
