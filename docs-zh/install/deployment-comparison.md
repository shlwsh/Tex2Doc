# Tex2Doc 生产环境部署方案对比分析报告

> **最后更新日期**：2026-06-26

在将 Tex2Doc 部署至生产环境时，根据客户端的实际需求、系统资源限制以及对转换质量（尤其是宏展开和动态链接交互性）的容忍度，目前系统支持两种级别的部署方案：**轻量化纯客户/终端转换方案（仅部署 Rust 服务）** 与 **高保真服务端全量部署方案（部署 Rust 服务 + TeX Live 运行时组件）**。

本文档详细对比两者的优劣势、转换质量差异及部署时的核心注意事项。

---

## 1. 方案对比概览

| 维度 | 方案 A：轻量化纯客户方案（仅 Rust 服务） | 方案 B：高保真服务端全量部署方案（含 TeX 组件） |
| :--- | :--- | :--- |
| **部署架构** | 仅部署编译好的 Rust `doc-server` 进程 + DB + 静态文件 | 在方案 A 的基础上，系统底座需安装 TeX Live（`xelatex` / `lualatex`） |
| **转换核心** | 依赖纯 Rust 内置的 `RuleBasedBackend` 解析 AST | 动态挂载 `XeLaTeXHookBackend` 或 `LuaTeXNodeBackend` 进行带钩子的编译 |
| **系统资源占用** | **极低**（内存 MB 级别，冷启动瞬间完成，无需庞大的运行时存储） | **高**（需要数 GB 的磁盘空间安装 TeX，运行时需拉起外部子进程，资源开销大） |
| **转换耗时** | **极快**（完全在内存中进行正则与规则匹配，无外接 IO 负担） | **较慢**（需生成临时文件、启动 TeX 编译器进程、解析输出事件及字节码） |
| **目标场景** | 客户端独立打包、对部署体积敏感的私有化边缘节点、基础文档快速预览 | 商业化核心转换 SaaS 集群、对排版保真度和文献跳转要求极高的严苛场景 |

---

## 2. 转换质量与能力差异分析

仅部署 Rust 服务由于缺少 TeX 编译器的动态宏展开能力，转换过程会触发引擎的**自动降级**。整个流程不会报错中断，但会在以下高阶功能上出现质量折损：

### 2.1 交叉引用与超链接（最显著差异）
- **方案 A（降级）**：由于无法捕捉真实的编译期宏执行事件，文章中的图表引用（`\ref`）和参考文献引用（`\cite`）会被静态语法树退化解析为**普通纯文本**（例如 `[1]` 或 `图 1.1`）。输出的 DOCX 中不支持双击跳转。
- **方案 B（全量）**：Rust 引擎会注入编译钩子（Hook），精确拦截并提取引用事件的宏展开映射。在输出的 DOCX 中，引用会生成原生的 **Word 书签（Bookmark）和超链接（Hyperlink）**，支持正文与参考文献间的无缝双向跳转，交互体验最佳。

### 2.2 极端自定义宏还原度
- **方案 A（降级）**：纯 Rust 解析器基于语法规则树（Rowan AST）。面对作者手写的极度复杂的底层宏控制流（例如重度依赖低级 TeX 计数器/寄存器或特殊条件判断的自建环境），可能存在解析精度下降甚至结构丢失的问题。
- **方案 B（全量）**：真正的 TeX 引擎会在后台负责动态计算与展开宏定义，能够实现 100% 完美的宏语义还原。

### 2.3 版面物理数据（LayoutGraph）
- **方案 A（降级）**：无法提取版面坐标数据（后台日志可见 `layout: None`）。对于标准的 DOCX 流式排版不会有太大影响，但如果业务后续规划了 PDF 视觉对齐比对、像素级字形辅助校验等高阶功能，这部分功能将处于失效状态。
- **方案 B（全量）**：通过提取 XDV 字节码（XeLaTeX）或直接读取排版节点树（LuaTeX Node Tree），能够收集精确到单个字符级别的高度、宽度、深度等细粒度物理版面坐标。

## 3. 部署优劣势与核心注意事项

### 方案 A：轻量化纯客户方案（仅部署 Rust 服务）

* **优势**：开箱即用，零系统依赖冲突；磁盘体积小巧（百 MB 以内）；执行速度极快，无需担心 TeX 编译链死循环耗尽服务器资源。适合嵌入桌面端、WebAssembly 前端及内网轻量主机。
* **劣势**：缺少高级语义映射与动态超链接交互；无法应对极少数语法异常诡异的高级宏。

**⚠️ 部署注意事项：**
1. **预期管理**：在客户侧独立部署或交付前，须向客户说明跨文献引用的降级表现，确保业务方可接受无 Word 超链接支持的排版。
2. **基础配置不可省**：虽然免去了 TeX 环境，依然需配置 Rust 服务的常规环境变量，如 `DATABASE_URL`（PostgreSQL 连接串）以及 `TEX2DOC_STATIC_DIR`（静态前端资源托管路径）。
3. **无缝兜底机制**：无需改动 Rust 服务的任何代码逻辑。默认配置下 `allow_backend_fallback = true`，遇到无 TeX 环境时会自动通过静默判定切入 Rule-based 路线。

### 方案 B：高保真服务端全量部署方案（含 TeX 组件）

* **优势**：提供最具竞争力的排版解析质量，生成文档 100% 保留原生链接及排版保真度，商业化服务的主流形态。
* **劣势**：容器化镜像过于庞大（动辄多达数 GB）；编译任务执行时会拉高宿主机的 CPU/IO 负载。

**⚠️ 部署注意事项：**
1. **依赖环境安装**：必须在 Linux 服务器或基础 Docker 镜像中安装 TeX Live 套件。建议明确包含 `texlive-xetex` 和 `texlive-luatex` 及其所依赖的关键字体与中英文字宏包。推荐直接使用现成的带有完整 TeX 环境的基础 Docker 镜像，减少包管理的各种坑。
2. **环境变量验证**：必须确保 `xelatex` 和 `lualatex` 可执行程序处于执行 `doc-server` 进程的操作用户的 `PATH` 环境变量下。系统底层的 `command_available()` 利用 PATH 定位后端，若未能识别仍会被迫降级到方案 A。
3. **磁盘挂载与清理策略**：
    * TeX 编译时会在 Temp 目录（系统级或自定义如 `.texlive-cache`、`__docx_semantic_events.jsonl` 等）产生诸多缓存或中间事件文件。须确保运行系统具备定期垃圾回收机制，避免磁盘长时间运行被塞满。
4. **性能隔离与并发控制（非常关键）**：
    * 由于调用 TeX 后端需要起动隔离的子进程（Subprocess），大规模高并发时会大幅增加宿主机的 CPU 与内存承载压力。
    * **强烈建议** 在业务网关层或内部任务调度层设定合理的任务转换队列（排队机制）与进程执行**超时阈值（Timeout）**，以防止死循环格式错误的异常 LaTeX 源文件长期占用核心态资源，造成 OOM 故障。
